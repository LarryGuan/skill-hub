# L1 方法论精华

## 核心方法：5 个为什么

从用户的表层需求出发，连续追问"为什么"，直到到达不可再分解的本质定义。

**操作要点**：
- 从用户描述出发，找到最表层需求
- 每次问完等用户回答，不要自问自答
- 到达"目的"（为什么做）而非"手段"（怎么做）时停止

**判断标准**：
- 定义还包含"怎么做" → 继续追问
- 定义是纯粹的"为什么" → 到达终点

## 核心方法：手段 vs 目的

- **手段**：如何做（功能、技术、流程） — 不属于 L1
- **目的**：为了什么（价值、效果、成果） — L1 的内容

**识别手段的特征词**："做一个""用XX技术""通过XX方式"
**识别目的的特征词**："为了""达到""实现XX效果"

## 核心方法：粒度校准

目的的粒度有三个层级：

| 粒度 | 示例 | 问题 |
|------|------|------|
| 太抽象 | 提升用户体验 | 无法指导设计，任何功能都能说"提升体验" |
| 适中 | 让用户 3 步内完成下单 | 具体可衡量，不限制实现方式 |
| 太具体 | 做一个搜索框带筛选按钮 | 限制了可能，这是功能不是目的 |

**校准方法**：如果目的包含具体 UI/功能描述 → 太具体，往上抽象一层。如果目的适用于任何产品 → 太抽象，往下具体化。

## 常见误区

1. **把功能当目的**："做一个搜索功能" → 目的应该是"让用户快速找到目标内容"
2. **目的太抽象**："提升转化率" → 缺少当前基线和目标值
3. **缺少验收标准**：目的是"提升效率"但没有说从多少提升到多少
4. **只有正向没有痛点**：只说"想做什么"不说"现在遇到了什么问题"

## 追问技巧

- 用户说解决方案（"我想做一个XX系统"）→ 追问："为什么需要这个系统？它要解决什么问题？"
- 痛点太抽象（"效率低"）→ 追问："具体低在哪里？能举个例子吗？"
- 只有正向没有痛点 → 追问："现在没有这个系统的时候，是怎么做的？有什么不方便？"
- 用户卡住 → 提供行业类比帮助思考，但标注"这是参考示例"
