# L1 质量检查清单

## 完整性检查

- [ ] 背景与痛点是否清晰？（有具体的当前问题，不是泛泛描述）
- [ ] 第一性原理定义是否是一句话本质描述？（不含手段）
- [ ] 核心特征是否有 3-4 个维度？（含核心能力和底线）
- [ ] 核心目的是否有 3-5 个？（且都是成果而非功能）
- [ ] 验收标准是否可衡量？（至少 3 个角色视角）
- [ ] 约束摘要是否精炼？（控制在 200 字以内）

## 质量检查

- [ ] 是否区分了手段与目的？（产出中不应出现具体功能描述）
- [ ] 目的粒度是否适中？
  - 太抽象："提升用户体验" → 要求具体化
  - 适中："让用户用最少的操作找到目标内容" → 通过
  - 太具体："做一个带按钮的搜索框" → 提醒这是手段
- [ ] 验收标准是否可验证？（有量化指标或明确判定条件）
- [ ] 核心特征是否覆盖了安全/质量底线？

## 常见错误自查

- [ ] 是否避免了"做一个XX功能"式的目的描述？
- [ ] 是否避免了"提升用户体验"这种无法指导设计的抽象描述？
- [ ] 是否避免了缺少验收标准的情况？
- [ ] 第一性原理定义是否真的不含手段？（检查是否包含"系统""平台""功能"等词）
- [ ] 痛点是否具体？（不是"效率低"，而是"查找文档平均耗时15分钟"）

## v2 框架检查

- [ ] 01-业务目的/CLAUDE.md 已生成？（约束摘要 + 下游须知）
- [ ] 项目 index.md 的 L1 进度标记已更新为 [x]？
- [ ] 项目根 CLAUDE.md 未被修改？（框架宪法，只读）

## 写入前

以上所有检查项必须通过。任何一项不通过都必须补全后才能写入文件。
