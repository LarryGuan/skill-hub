# L1 产出格式模板

## 文件一：01-业务目的/index.md

### 标准格式

```markdown
# 业务目的

## 约束摘要（供下层快速引用）
（本层为顶层，无上层约束。以下为本层核心结论摘要）
- 本质：[第一性原理定义]
- 核心目的：[3-5个目的的一句话概括，用逗号分隔]
- 核心约束：[从核心特征和验收标准提炼的2-3条关键底线]

## 背景与痛点

### 当前问题
- [问题1]
- [问题2]
- [问题3]

### 为什么需要解决
- [原因1]
- [原因2]

### 时机为什么是现在
- [外部因素]
- [内部因素]

## 业务方向

### 第一性原理定义
[一句话本质定义]

### 核心特征
| 维度 | 说明 |
|------|------|
| [维度1] | [说明] |
| [维度2] | [说明] |
| [维度3] | [说明] |

## 核心目的
1. **[目的名]**：[具体描述]
2. **[目的名]**：[具体描述]
3. **[目的名]**：[具体描述]

## 验收标准
| 角色 | 验收标准 |
|------|----------|
| 研发 | [标准] |
| 产品 | [标准] |
| 运营 | [标准] |
```

### 字段说明

| 字段 | 要求 | 常见错误 |
|------|------|----------|
| 约束摘要 | 控制在 200 字以内，供 L2-L5 快速加载 | 太详细（变全文摘要）或太简略（丢失关键约束） |
| 当前问题 | 具体可感知的痛点，2-5 条 | 太抽象（"效率低"）或太少（只有 1 条） |
| 第一性原理定义 | 一句话，不含手段 | 包含"做一个XX系统"（手段） |
| 核心特征 | 3-4 个维度，覆盖能力和底线 | 太多维度（超过 5 个）或缺少底线 |
| 核心目的 | 3-5 个，每个是成果而非功能 | "做一个XX功能"（手段）或"提升体验"（太抽象） |
| 验收标准 | 至少 3 个角色视角，可量化 | 缺少运营视角或标准不可衡量 |

### 约束摘要写作规则

- 本质：从第一性原理定义中提炼，不超过 30 字
- 核心目的：用逗号分隔的短语列表
- 核心约束：从特征和验收标准中选最关键的 2-3 条底线

## 文件二：01-业务目的/CLAUDE.md

### 标准格式

```markdown
# L1 约束摘要
{200字以内：第一性原理、核心目的、验收标准}

## 下游须知
- 核心目的（L2 的"做什么"必须服务于这些）
- 验收标准（L5 的功能规格必须能覆盖这些）
```

### 编写规则

| 字段 | 规则 |
|------|------|
| 约束摘要 | 从 index.md 的约束摘要章节提取，200 字以内 |
| 下游须知 | 列出 L2 和 L5 必须注意的核心目的和验收标准 |
