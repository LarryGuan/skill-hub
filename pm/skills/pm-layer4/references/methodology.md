# L4 方法论精华

## 核心方法：STAR 旅程画法

从 L2 关系事件出发，用 STAR 结构画用户旅程。

### STAR 结构

| 要素 | 说明 | 示例 |
|------|------|------|
| Situation（触发条件） | 什么情况下启动 | 买家提交订单后，等待卖家确认发货 |
| Task（用户目标） | 用户想达成什么 | 卖家要确认订单并安排发货 |
| Action（步骤） | 用户按时间顺序的操作 | 见下方步骤结构 |
| Result（结束状态） | 场景结束后用户和系统各处于什么状态 | 卖家已确认，订单进入"待发货"状态 |

### 步骤结构

每个 Action 步骤包含三个标注：

| 标注 | 说明 |
|------|------|
| 用户操作 | 用户做了什么（从 L3 画像的行为模式推导） |
| 系统接触点 | 🔴 需要系统支撑 / ⚪ 用户独立完成 |
| 痛点命中 | 如果命中了 L3 的某个痛点，标注引用（如"P1: 查找供应商耗时"） |

### 异常路径标注

每个旅程标注关键决策点和异常分支：

```
步骤3：系统处理订单
    ↓
【决策点】库存充足？
    ├─ 是 → 继续正常流程
    └─ 否 → 走补货/通知流程（异常场景）
```

## 核心方法：系统接触点标注

### 两类接触点

- **🔴 需要系统支撑**：用户这一步必须依赖系统才能完成（如查询数据库、发送通知、计算价格）
- **⚪ 用户独立完成**：用户不需要系统就能完成（如思考、手动检查、线下沟通）

### 场景筛选规则

只写"依赖系统"的场景：
- 🔴 接触点密集的旅程段才提炼为场景故事
- 纯 ⚪ 的段落不需要场景故事（用户自己做就行）
- 一个旅程可能拆出多个场景（正常流程 + 异常分支各自独立）

---

## 核心方法：场景故事编写

### 编写步骤

1. 从原始旅程中找出 🔴 接触点密集的段落
2. 将这些段落整理为结构化场景故事
3. 标注场景分类（核心/边缘/异常）
4. 编写活动图

### 场景分类

| 分类 | 说明 | 处理方式 |
|------|------|---------|
| 核心场景 | 必须支持，体现核心价值 | MVP 必须实现 |
| 边缘场景 | 可后续迭代 | 记录但可延后 |
| 异常场景 | 验证重点 | 必须有处理方案 |

### 场景故事写作技巧

**技巧一：具体化，避免抽象**

❌ 抽象：用户咨询问题，系统回复

✅ 具体：用户发送"根管治疗后有点肿，正常吗？"，系统识别到"肿"为敏感词，切换为草稿模式，模拟打字后等待人工确认

**技巧二：用"业务意图"收尾**

每个场景故事最后，说明这个场景的业务价值：
```
业务意图：展示"人机协作"如何保障医疗安全
```

**技巧三：突出关键决策点**

在场景中标注关键决策点，用分支结构展示异常路径。

**技巧四：场景要可验证**

好的场景故事应该：
- 让研发说"我知道怎么实现"
- 让测试说"我知道怎么验证"
- 包含明确的触发条件、操作步骤、系统响应、结束状态

## 核心方法：粒度控制

### 场景数量

每个终端 3-7 个核心场景，边缘和异常场景作为补充。不是越多越好。

### 粒度判断规则

- **拆分信号**：旅程超过 10 个步骤 / 存在 2 个以上独立决策点 / 触发条件完全不同 → 拆为多个场景
- **合并信号**：两个场景共享同一触发条件和同一参与者 / 其中一个场景不足 3 步 → 考虑合并
- **异常场景独立**：异常分支永远独立为一个场景，不与正常流程合并

## 核心方法：交叉分析

### 三维度分析

| 维度 | 说明 | 典型问题 |
|------|------|---------|
| 依赖关系 | 场景间的先后依赖 | A 的后置条件是 B 的前置条件吗？ |
| 冲突检测 | 同一数据/资源在不同终端场景中的操作冲突 | 两个终端同时修改同一条数据怎么办？ |
| 数据流 | 场景间的数据传递路径 | A 场景产出的数据，B 场景怎么消费？ |

### 分析要点

- 依赖关系：标注哪些场景必须先完成
- 冲突检测：识别并发操作的数据竞争
- 数据流：画出场景间的数据传递链路

## 核心方法：用例图细化

### 5 步转换规则

1. **基线**：从 L2 用例图初版出发（角色 + 用例 + 系统边界已定）
2. **细化交互**：场景旅程中的每个 🔴 系统接触点 → 对应用例的一个交互步骤
3. **扩展用例**：异常场景中的异常分支 → 新增为对应用例的 `<<extend>>` 用例
4. **补充用例**：如场景中出现了 L2 初版未覆盖的系统行为 → 新增用例
5. **验证**：所有纳入事件都有对应场景 → 所有场景都有对应用例

### 与 L2 初版的关系

用例图细化版不是重画，是在 L2 初版基础上：
- 已有角色不变
- 已有用例细化交互步骤
- 新增异常分支用例（extend 关系）
- 补充场景中发现的新用例

## 核心方法：活动图画法

### 画法规则

- **泳道** = 参与者（从 L3 画像取，不是系统组件）
- **节点** = 用户操作 + 系统响应
- **决策点** = 旅程中的关键判断（菱形节点）
- **分支** = 正常路径 + 异常路径

### 注意事项

- 活动图反映用户操作路径，不反映系统内部逻辑
- 每个泳道只放该参与者的操作
- 决策点要与场景故事中的决策点对应

## 常见误区

### 误区一：场景写成功能说明

❌ 错误：场景描述"系统有一个搜索功能"

✅ 正确：场景描述"用户想找某项服务，通过搜索找到"

### 误区二：场景太理想化

❌ 错误：用户发送消息，系统立即回复完美答案

✅ 正确：用户发送消息，系统处理 3 秒后回复，可能需要用户澄清

### 误区三：忽略异常场景

❌ 错误：只描述正常流程

✅ 正确：同时描述正常和异常场景（如：敏感词拦截、知识库无匹配）

### 误区四：场景与功能混淆

- **场景**：描述"发生了什么"（用户视角）
- **功能**：描述"怎么实现"（系统视角）

场景是功能的输入，功能是场景的支撑。

## 追问技巧

- 用户只说"用户用一下" → 追问："用户第一步做什么？打开什么？"
- 用户说"系统处理一下" → 追问："系统具体做什么？需要几秒？用户在等什么？"
- 用户说"如果出错了" → 追问："什么算出错？用户看到什么？能重试吗？"
- 用户跳过细节 → 追问："这一步用户最在意什么？有没有 L3 痛点被命中？"
- 用户描述功能而非场景 → 追问："用户为什么要用这个功能？他想达成什么？"
