# L4 质量检查清单

## 完整性检查

- [ ] 所有 L2 纳入事件都有对应场景或合理的排除说明？（逐事件核对）
- [ ] 所有 L3 主体都在场景中作为参与者出现？（不能遗漏间接主体）
- [ ] L2 终端分工表中每个终端都有场景目录？（即使只有 1 个终端也要按终端组织）
- [ ] 每个场景故事要素完整？（基本信息、用户旅程、决策点、设备环境、业务意图、活动图）
- [ ] 交叉分析三维度齐全？（依赖关系、冲突检测、数据流）
- [ ] 用例图细化版覆盖所有场景？（与 L2 初版衔接）
- [ ] 活动图覆盖核心流程的用户操作路径？（泳道=参与者，含决策点）
- [ ] 04-用户场景/ 按终端分目录？（每个终端下有 scenario 文件）

## 场景质量检查

- [ ] 场景回答"在什么情况下做什么"？（不是纯功能清单，有具体触发条件和操作描述）
- [ ] 场景分类已标注？（核心/边缘/异常有区分，不是全标核心）
- [ ] 场景粒度适中？（每个场景聚焦一个关系事件，不过粗也不过细）
- [ ] 设备环境与 L3 使用环境一致？（终端类型、网络、能力约束对得上）
- [ ] 参与者行为与 L3 画像心理特征匹配？（不矛盾）
- [ ] 场景故事可验证？（研发能理解怎么实现，测试能理解怎么验证）

## 回溯检查

- [ ] L2 终端场景清单中的事件在 L4 场景中都有对应？（终端场景清单覆盖）
- [ ] L3 痛点列表中每个核心痛点在至少一个场景中被"痛点命中"引用？
- [ ] L3 使用环境与场景中的设备环境描述一致？
- [ ] L2 边界表"不纳入"的事件不出现在 L4 场景中？

## 框架检查

- [ ] 04-用户场景/CLAUDE.md 已生成？（约束摘要 + ⚠️ 强制回溯项 + 下游须知）
- [ ] 04-用户场景/CLAUDE.md 的约束摘要控制在 200 字以内？
- [ ] 04-用户场景/CLAUDE.md 的强制回溯项包含场景故事列表 + 交叉分析？
- [ ] 项目 index.md 的 L4 进度标记已更新为 [x]？
- [ ] 项目根 CLAUDE.md 未被修改？（框架宪法，只读）

## 常见错误自查

- [ ] 是否避免了场景写成功能说明？（场景描述"用户做了什么"，不是"系统有什么功能"）
- [ ] 是否避免了场景太理想化？（考虑了延迟、错误、用户困惑）
- [ ] 是否避免了忽略异常场景？（核心场景都有异常分支）
- [ ] 是否避免了场景与功能混淆？（场景是用户视角，功能是系统视角）
- [ ] 是否避免了产品上下文新增维度？（L4 只引用 L3 完成版，不新增）

## 写入前

以上所有检查项必须通过。任何一项不通过都必须补全后才能写入文件。
