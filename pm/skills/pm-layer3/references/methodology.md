# L3 方法论精华

## 核心方法：主体→画像

从 L2 的粗粒度主体推导具体的用户画像。

### 画像构建要素

| 要素 | 说明 | 示例 |
|------|------|------|
| 角色定位 | 在业务中扮演什么角色 | 核心采购决策者 |
| 典型特征 | 年龄/职业/技术水平（帮助具象化） | 30-45岁，制造业采购，PC熟练 |
| 核心目标 | 使用产品要达成的成果（不是功能） | 快速找到合格供应商并完成比价 |
| 动机 | 为什么要达成这个目的 | 年底采购预算要控制，货比三家不吃亏 |
| 行为模式 | 通常怎么完成相关任务 | 先在 Excel 里整理需求，再逐个联系供应商 |
| 心理特征 | 在意什么、怕什么、偏好什么 | 风险厌恶，偏好有历史合作的供应商 |

### 画像构建步骤

1. 从 L2 主体链接图中提取所有主体
2. 对每个主体，结合 L1 核心目的思考：这个主体在产品中要达成什么
3. 引导用户补充真实信息（不编造画像特征）
4. 确保画像与 L2 业务事件一致（如果主体在 L2 只参与"接单"，画像不应该侧重"管理"）

### 画像粒度控制

- **不要**：过于精确的人口统计学描述（"32岁，本科，月薪1.5万"）
- **要**：帮助理解行为动机的关键特征（"价格敏感，不喜欢等待，习惯手机操作"）
- 画像是设计参考，不是市场调研报告

## 核心方法：痛点挖掘

### 从画像到痛点

痛点的来源有三个：

1. **行为模式痛点**：当前做法的效率或体验问题
   - 画像："习惯先在 Excel 整理需求" → 痛点："手动录入需求容易出错，格式不统一"
2. **心理特征痛点**：用户的担忧或不满
   - 画像："风险厌恶" → 痛点："不敢尝试新供应商，怕质量不稳定"
3. **环境痛点**：使用场景中的客观限制
   - 画像："在仓库现场操作" → 痛点："仓库信号差，页面加载慢"

### 痛点质量标准

| 标准 | 合格 | 不合格 |
|------|------|--------|
| 具体 | "查找供应商平均要翻 3 页结果" | "查找不方便" |
| 可感知 | "每次都要手动复制粘贴到 Excel" | "效率低" |
| 有关联 | 关联到 L2 事件"买家搜索供应商" | 孤立的投诉，不知道发生在哪个环节 |
| 有影响 | "导致错过最优报价" | 无后果描述 |

### 痛点与 L2 事件的关联

每个痛点必须能追溯到 L2 主体链接图上的具体事件：

| 痛点 | 关联的 L2 事件 | 严重程度 |
|------|---------------|---------|
| 查找供应商耗时 | 买家→平台："1.搜索供应商" | 核心 |
| 比价困难 | 买家→平台："2.获取报价" | 核心 |
| 订单状态不透明 | 买家→平台："5.查看订单状态" | 次要 |

关联 L2 事件的好处：
- L4 可以从事件→痛点推导场景中的用户情绪
- L5 可以从痛点推导功能需求
- 形成完整追溯链：痛点→事件→场景→功能

### 痛点严重程度判断

- **核心痛点**：如果不解决，用户不会使用或频繁流失
- **次要痛点**：解决后体验提升明显，但不解决也不致命

## 核心方法：使用环境描述

### 从终端分工到使用环境

L2 的终端分工表已经定义了每个主体用什么终端。L3 要把终端变成真实的使用环境：

```
L2 终端分工：买家 → PC网页
    ↓ L3 推导
使用环境：办公室台式机，双屏，稳定网络，工作时间段集中使用，可被打断
```

### 环境描述维度

| 维度 | 说明 | 典型问题 |
|------|------|---------|
| 设备 | 终端类型、操作系统、屏幕尺寸 | "PC网页是在笔记本还是台式机上？" |
| 场景 | 物理环境 | "在办公室还是在外面？" |
| 网络 | 稳定性、带宽 | "4G 还是 WiFi？会不会断网？" |
| 时间 | 使用频率和时长 | "每天用几次？每次多久？" |
| 干扰 | 可能被打断的情况 | "会不会被电话打断？" |
| 能力 | 操作水平 | "技术小白还是熟练用户？" |

### 环境约束对设计的影响

| 环境约束 | 对 L4/L5 的设计影响 |
|---------|-------------------|
| 仓库信号差 | 页面要轻量，支持离线或弱网 |
| 频繁短时间使用 | 操作要快捷，减少步骤 |
| 可能被打断 | 支持中断恢复，自动保存 |
| 触屏操作 | 按钮要大，避免精确点击 |
| 非技术用户 | 界面要直观，减少学习成本 |

## 核心方法：产品上下文积累

### L1→L2→L3 的积累关系

```
L1 种子：为什么做（第一性原理、核心目的、验收标准）
    ↓ + L2 扩展
L2 扩展：做什么不做什么（边界、终端、业务闭环）
    ↓ + L3 完成
L3 完成：为谁做、什么环境（用户画像、痛点、使用环境）
```

### 产品上下文完成版的结构

产品上下文完成版包含三个维度：
1. **为什么做**（来自 L1）：本质定义、核心目的
2. **做什么不做什么**（来自 L2）：关键边界、终端架构、业务闭环
3. **为谁做**（来自 L3）：核心用户、核心痛点、使用环境

### 积累规则

- L3 完成后，产品上下文不再新增维度
- L4/L5 只引用产品上下文，不再修改
- 产品上下文完成版是 L4/L5 的必读输入

## 常见误区

1. **画像太详细**：变成市场调研报告 → 保持简洁，聚焦影响设计的特征
2. **痛点太抽象**："体验不好" → 必须具体可感知
3. **痛点脱离业务**：与 L2 事件无关联 → 每个痛点必须关联到 L2 事件
4. **环境描述脱离终端**：L2 说用手机，L3 却描述 PC 环境 → 与终端分工表对齐
5. **遗漏主体**：L2 有 4 个主体，L3 只画了 2 个画像 → 所有主体都要有画像
6. **产品上下文增加新维度**：L3 之后还加"怎么赚钱" → 这属于 L2 的范围，L3 只加"为谁做"

## 追问技巧

- 用户说"我们的用户是企业" → 追问："是企业里什么角色？采购？财务？老板？"
- 用户说"体验不好" → 追问："具体哪里不好？能举个例子吗？"
- 用户说"都是年轻人" → 追问："这意味着什么？技术接受度高？喜欢尝鲜？"
- 用户只说功能需求 → 追问："这是哪个角色的需求？他们现在是怎么做的？"
- 用户说"在手机上用" → 追问："在什么场景下？走路时？坐地铁时？信号怎么样？"
