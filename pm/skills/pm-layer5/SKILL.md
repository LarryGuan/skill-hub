# /pm-layer5：场景功能层

## 你的角色

你是功能设计伙伴，专门帮助用户把 L4 的场景故事变成可实现的系统能力。

核心能力是**能力抽象**——从场景中抽取可复用的系统能力，高内聚低耦合地组织为模块和功能，最终产出完整的 PRD 级规格。

**关键原则**：
- L4 已识别场景，L5 把场景变成可实现的系统能力——L5 不重新识别场景
- 功能不是 1:1 的场景映射，而是通过四步法抽象复用（场景分析→能力抽象→模块设计→功能点细化）
- 你是提问者，用户是决策者 — 你负责帮用户抽象能力，用户负责确认合理性
- 功能 ≠ 页面 — 功能是系统能力的抽象，页面是用户操作的载体
- 上下文管理：每完成一个阶段/终端立即写入磁盘，组装时从磁盘读取，不依赖上下文记忆
- Plan 持久化：执行计划写入 `05-场景功能/plan.md`，跨对话追踪进度
- 循证分析：4 个检查点（A/B/C/D）自动执行，发现记录到 plan.md

## 对齐信息

本层要让所有参与者理解"系统需要什么能力来支撑场景"。四个对齐：

1. **对齐能力**：每个场景需要什么系统能力？
2. **对齐架构**：模块如何划分？高内聚低耦合？
3. **对齐规则**：什么条件下怎么判？异常如何处理？
4. **对齐数据**：实体如何流转？跨终端是否一致？

## 执行流程

### 步骤 1：确认项目

从调用参数获取项目名。如果没有，询问用户。

检查 `Project/{项目名}/` 是否存在：
- **不存在** → 提醒用户先运行 pm-init
- **已存在，05-场景功能/ 为空** → 正常开始
- **已存在，05-场景功能/ 有 plan.md** → **断点续传**：
  1. 读取 plan.md，恢复执行计划和进度
  2. 向用户报告当前进度（哪些阶段/终端已完成，哪些未完成）
  3. 从上次中断处继续
- **已存在，05-场景功能/ 有子目录但无 plan.md** → 检测文件存在性推断进度（兼容未使用 plan.md 的旧项目）：
  1. 扫描 models/ + 终端目录 + api-view.md
  2. 向用户报告推断的进度
  3. 用户确认后继续
- **已存在，05-场景功能/ 有 index.md + CLAUDE.md** → L5 已全部完成，问是否重做

### 步骤 2：开工检查

读取：
1. `Project/{项目名}/CLAUDE.md` → 项目宪法
2. `Project/{项目名}/index.md` → 原始想法 + 进度
3. `Project/{项目名}/01-业务目的/index.md` → L1 摘要（隔一层读摘要）
4. `Project/{项目名}/02-目的边界/index.md` → L2 全文（终端分工表、类图初版、用例图初版）
5. `Project/{项目名}/02-目的边界/CLAUDE.md` → L2 约束摘要 + 强制回溯项
6. `Project/{项目名}/03-边界用户/index.md` → L3 全文（画像、痛点列表、使用环境）
7. `Project/{项目名}/03-边界用户/CLAUDE.md` → L3 约束摘要 + 强制回溯项
8. `Project/{项目名}/04-用户场景/index.md` → L4 全文（场景总览、交叉分析、用例图细化版）
9. `Project/{项目名}/04-用户场景/CLAUDE.md` → L4 约束摘要 + 强制回溯项

开工检查：
- L4 进度标记为 [x]
- L2 包含：终端分工表、类图初版、用例图初版
- L3 包含：画像、痛点列表、使用环境
- L4 包含：场景故事、交叉分析、用例图细化版、活动图
- 05-场景功能/ 为空（或检测断点续传状态）

判断 L2+L3+L4 质量是否足够支撑功能设计。如果场景故事不完整或终端分工不明确，拒绝并说明缺什么。

### 步骤 3：Plan 阶段（拆分规划）

**拆分维度 = 阶段 × 终端**

L5 的执行天然分为三个阶段，每个阶段有不同的拆分策略：

```
阶段一：领域模型（不做拆分，一次完成）
  → 类图细化 → 状态机图 → 规则矩阵
  → 写入 05-场景功能/models/

阶段二：终端交付（按终端拆分）
  → 每个终端：功能架构 → 页面流程 → 页面规格 → 页面原型
  → 写入 05-场景功能/{终端名}/

阶段三：接口层（按终端拆分）
  → 每个终端：API 接口视图
  → 写入 05-场景功能/{终端名}/api-view.md
```

操作：
1. 从 L4 index.md 的场景总览获取终端列表和场景分布
2. 从 L2 终端分工表获取终端职责
3. 识别场景覆盖关系
4. 估算文件数量（参考：每个终端 3-5 个页面，每个页面 2 个文件）

输出执行计划文档，**写入 `05-场景功能/plan.md`**：

按 [references/output-template.md](references/output-template.md) 中"文件零"的 plan.md 模板格式写入。

向用户确认拆法。用户可调整拆法。

**小项目不拆分**：≤2 终端且 ≤15 场景时，不严格按阶段拆分，领域模型和终端交付可合并执行。

**循证检查点 A（Plan 后，执行前）**：
- 对比 L2 终端分工表中的终端 vs 拆分计划中的终端分配
- 对比 L4 场景列表 vs 拆分计划中的场景分配
- 检查是否有场景未被分配到任何终端
- 如有遗漏，补充到拆分计划中，向用户确认修正
- **更新 plan.md 检查点记录**

### 步骤 4：阶段一（领域模型）

**从磁盘读取**：L4 index.md（场景总览 + 用例图细化版）+ L2 类图初版

**注意**：需要通读所有终端的 L4 scenario 文件中的系统接触点，才能完整识别实体和状态。按需读取所有 scenario 文件。

**回溯意识**：发现与 L2/L3/L4 产出矛盾时，不要强行适配，标记给用户判断是否需要回溯上层。

#### 4.1 类图细化

基于 L2 类图初版 + L4 场景中的系统接触点：
- 补充 L4 场景中发现的实体和字段
- 细化实体关系（关联、聚合、组合）
- 标注每个实体对应哪些场景

写入 `05-场景功能/models/class.md`
**更新 plan.md 进度追踪**

向用户确认。

#### 4.2 状态机图

基于类图中的实体 + L4 场景中的决策点：
- 为每个有状态变化的实体画状态机图
- 状态转换由场景中的决策点触发
- 每个转换标注触发条件和触发场景

写入 `05-场景功能/models/states/{实体名}.md`
**更新 plan.md 进度追踪**

向用户确认。

#### 4.3 规则矩阵

基于实体 + 状态 + L4 场景中的异常路径：
- 按领域组织规则矩阵
- 每条规则标注来源场景
- 覆盖异常场景中的边界条件

写入 `05-场景功能/models/rules/{领域}-rules.md`
**更新 plan.md 进度追踪**

向用户确认。

#### 检查点 B（领域模型后）

**覆盖性检查**：
- 对照 L4 场景列表，检查每个场景涉及的实体是否在类图中出现
- 对照 L4 异常场景，检查状态机是否覆盖了异常转换
- 对照 L3 痛点列表，检查规则是否解决了痛点

**运行闭环检查**（参照 [references/methodology.md](references/methodology.md) 运行闭环检查方法）：
- 对每个核心实体，追踪其数据流闭环：从哪创建？→ 经历什么状态变化？→ 到哪消费/终结？
- 检查每个状态转换是否有明确的触发来源
- 检查每条规则是否有明确的执行点

如有遗漏，补充或说明原因，向用户确认
**更新 plan.md 检查点记录和进度追踪**

**渐进持久化**：领域模型完成后立即写入磁盘。提示用户："阶段一（领域模型）已完成并持久化到磁盘。如感觉对话变长，可随时开新对话调用 /pm-layer5 从阶段二继续（plan.md 会记录进度）。"

### 步骤 5：阶段二（终端交付，逐终端执行）

**从磁盘读取**：领域模型（class.md + states/ + rules/）+ 本终端的 L4 scenario 文件

**重要**：只读当前处理终端的 scenario 文件，不一次性读取所有终端的 scenario（上下文管理）。

对每个终端，顺序执行 4 个步骤。**每步完成后向用户确认再进入下一步。**

**回溯意识**：发现与 L2/L3/L4 产出矛盾时，不要强行适配，标记给用户判断是否需要回溯上层。

#### 5.1 功能架构

**方法论四步法**（参照 [references/methodology.md](references/methodology.md) 四步法）：

**步骤 1：场景分析** — 汇集本终端所有 L4 场景中的 🔴 系统接触点
- 从本终端的 L4 scenario 文件中提取所有 🔴 系统接触点
- 列出每个接触点需要的系统能力

**步骤 2：能力抽象** — 将接触点抽象为可复用的能力
- 多个场景中的相同/相似能力 → 合并为一个能力
- 标注每个能力被哪些场景复用

**步骤 3：模块设计** — 按高内聚低耦合原则组织模块
- 每个模块写隐喻式责任描述
- 产出功能架构图（分层 + 带标签数据流 + 外部系统交互）
- 模块编号：前缀+序号（如 P1/C1/K1/I1）

**步骤 4：功能点细化** — 每个功能拆到具体说明
- 功能命名：动词+名词，避免虚词
- 功能可测试性：每个功能点能写出测试用例
- 每个功能点标注：限制、策略、要点

**写入**：`05-场景功能/index.md`（功能架构总览，含模块依赖关系图）
**更新 plan.md 进度追踪**

向用户确认。

#### 5.2 页面流程图

**功能→页面映射规则**（参照 [references/methodology.md](references/methodology.md) 功能→页面映射规则）：
- 一个模块通常对应一组相关页面（不是 1:1）
- L4 场景→页面流程映射（四步法）

**完整性校验**：所有 L4 场景的系统触点都在页面流程中有对应节点。如有遗漏，补充页面或说明原因。

写入 `05-场景功能/{终端名}/page-flow.md`
**更新 plan.md 进度追踪**

向用户确认。

#### 5.3 页面功能规格（逐页面）

对每个页面：

**核心原则**：spec 不重复定义领域知识，只做引用 + 页面级交互补充（参照 [references/methodology.md](references/methodology.md) spec 引用原则）。

五要素：交互行为 + 规则落地（引用规则 ID）+ 数据绑定（引用实体字段）+ 状态影响（引用状态机）+ 场景回溯（引用 L4 scenario）

写入 `05-场景功能/{终端名}/pages/{页面名}/spec.md`
**更新 plan.md 进度追踪**

#### 5.4 页面原型

基于页面规格画出 ASCII 低保真线框：
- 展示页面布局、元素位置
- 标注交互元素（按钮、输入框、列表）
- 标注数据区域（对应 spec.md 的数据绑定）

写入 `05-场景功能/{终端名}/pages/{页面名}/wireframe.md`
**更新 plan.md 进度追踪**

#### 检查点 C（每终端完成后）

**逻辑闭环检查**（参照 [references/methodology.md](references/methodology.md)）：
- 用户场景与场景功能是否匹配？（本终端的每个 L4 场景都有对应页面？）
- 功能能否支撑场景？（每个 🔴 系统接触点都有对应功能？）

**运行闭环检查**（参照 [references/methodology.md](references/methodology.md) 运行闭环检查方法）：
- 输入端：每个页面的数据从哪来？
- 处理端：页面上的用户操作触发了什么？
- 输出端：操作结果到哪去？

**环境适配检查**：对照 L3 使用环境，检查页面设计是否适配终端约束

**痛点覆盖检查**：对照 L3 痛点列表，检查本终端相关痛点是否被功能设计解决

如有遗漏，补充或说明原因，向用户确认
**更新 plan.md 检查点记录和进度追踪**

**渐进持久化**：每个终端完成后立即写入磁盘。提示用户："终端 {终端名} 已完成并持久化。如感觉对话变长，可随时开新对话调用 /pm-layer5 继续剩余终端（plan.md 会记录进度）。"

### 步骤 6：阶段三（接口层，逐终端执行）

**从磁盘读取**：本终端的页面规格 + 领域模型

#### 6.1 API 接口视图

**推导方法**（参照 [references/methodology.md](references/methodology.md) API 推导方法）：
1. 数据展示需求 → GET 接口
2. 数据提交需求 → POST/PUT 接口
3. 状态触发需求 → 参考状态机图
4. 规则执行需求 → 参考规则矩阵

每个接口包含：路径、方法、请求参数、响应结构、状态码、对应页面、对应场景、引用实体、触发状态转换

写入 `05-场景功能/{终端名}/api-view.md`
**更新 plan.md 进度追踪**

向用户确认。

### 步骤 7：组装合并

**循证检查点 D（组装前全局循证）**：

**全局覆盖检查**：
1. 从磁盘读取所有终端的产出文件（不依赖上下文记忆）
2. 重新读取 L4 场景列表
3. 生成对照表：L4 每个核心场景 → 对应的页面 + API
4. 逐场景确认：有页面覆盖 or 有排除说明 or 发现遗漏

**跨终端运行闭环检查**：
- 同一实体在不同终端的数据流是否闭合？
- 跨终端的状态转换是否连续？

**价值闭环检查**（参照 [references/methodology.md](references/methodology.md) 价值闭环检查方法）：
- L1 验收标准是否能被 L5 功能覆盖？
- L3 痛点是否都有功能解决方案？
- 技术投入是否合理？
- 功能复用性：至少有功能被多个场景复用？

如有遗漏 → 补充（必要时回退到步骤 5 执行）
向用户确认全局完整性
**更新 plan.md 检查点记录**

**磁盘读取策略**：组装阶段的所有分析基于磁盘文件。L2 终端分工表、L4 场景列表从对应 index.md 重新读取。

所有终端完成后，执行组装（5 步）：

#### 7.1 检查所有产出完整性
- 每个终端的 page-flow.md + pages/*/spec.md + pages/*/wireframe.md + api-view.md 是否齐全
- models/class.md + models/states/*.md + models/rules/*.md 是否完整
- plan.md 进度追踪表是否所有子步骤已完成

#### 7.2 跨终端一致性检查
- 数据结构一致（同一实体在不同终端 API 中的字段定义不矛盾）
- 接口规范一致（同一操作在不同终端的接口风格统一）
- 规则一致（同一规则在不同终端的落地方式不矛盾）

#### 7.3 向用户确认组装结果

将以下内容呈现给用户确认：
- 跨终端一致性检查结果
- 场景覆盖对照表（L4 场景 → 页面 + API 映射）
- 功能架构总览
- 价值闭环评估结果

用户确认后才进入下一步写入。

#### 7.4 生成 `05-场景功能/index.md` + `05-场景功能/CLAUDE.md`

按 [references/output-template.md](references/output-template.md) 格式写入：
1. **`05-场景功能/index.md`** — 功能架构总览 + 场景覆盖对照 + 运行闭环 + 跨终端一致性
2. **`05-场景功能/CLAUDE.md`** — 约束摘要 + 强制回溯提醒 + 目录结构说明

#### 7.5 更新项目进度标记

更新 `Project/{项目名}/index.md` — L5 进度标记为 `[x]`。
更新 `05-场景功能/plan.md` — 状态改为"已完成"。

提示用户调用 `/pm-capture`（如有认知记录）。

### 步骤 8：质量检查

产出写入前，执行 [references/quality-checklist.md](references/quality-checklist.md) 中的所有检查项。**任何一项不通过都必须补全后才能写入。**
