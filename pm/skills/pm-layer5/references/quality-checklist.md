# L5 质量检查清单

## 完整性检查

- [ ] 所有 L4 核心场景都有对应页面 + 功能规格 or 合理排除说明？（逐场景核对）
- [ ] 所有 L3 主体都有对应终端的功能覆盖？
- [ ] 类图细化版覆盖所有场景中的实体？（对照 L4 scenario 中的实体）
- [ ] 状态机覆盖所有有状态变化的实体？（对照 class.md 中有状态标记的实体）
- [ ] 规则矩阵覆盖所有异常场景的边界条件？（对照 L4 异常场景）
- [ ] 每个页面规格要素齐全（交互行为、规则落地、数据绑定、状态影响、场景回溯）？
- [ ] 每个页面原型标注了交互元素和数据区域？
- [ ] API 接口视图覆盖所有页面的数据需求？（对照每个 spec.md 的数据绑定）
- [ ] 05-场景功能/ 按终端分目录 + models/ 共享？
- [ ] 每个终端目录下有：page-flow.md + api-view.md + pages/？
- [ ] 每个页面有：spec.md（功能规格）+ wireframe.md（原型）？
- [ ] plan.md 进度追踪表所有子步骤已完成？

## 功能质量检查

- [ ] 功能命名使用动词+名词，无虚词？（❌ 智能优化 → ✅ 消息去重）
- [ ] 每个功能点有可测试性？（能写出测试用例）
- [ ] 功能架构符合高内聚低耦合？（模块内功能相关，模块间依赖最小）
- [ ] 模块编号规范统一？（前缀+序号，如 P1/C1/K1/I1）
- [ ] 每个模块有隐喻式责任描述？（非技术人员可秒懂）
- [ ] 功能架构图分层排列 + 带标签数据流 + 外部系统交互？
- [ ] 至少有功能被多个场景复用？（不是 1:1 场景→功能映射）
- [ ] spec.md 不重复定义领域知识？（只引用 + 页面级交互补充）

## 回溯检查

- [ ] L4 场景列表中每个核心场景在 L5 有对应页面 + API？（场景覆盖）
- [ ] L2 终端分工表中每个终端的页面差异与之一致？（终端一致性）
- [ ] L3 痛点列表中每个核心痛点被至少一条规则或功能解决？（痛点覆盖）
- [ ] L2 类图初版中的实体在 L5 类图细化版中都有体现？（类图连续性）
- [ ] L4 用例图细化版中的用例在 L5 功能架构中有对应功能？（用例→功能覆盖）
- [ ] 跨终端数据结构一致？（同一实体在不同终端 API 中的字段定义不矛盾）

## 价值闭环检查

- [ ] L1 验收标准覆盖：L1 的每条验收标准都有对应的 L5 功能支撑？（对照 L1 index.md）
- [ ] 痛点解决覆盖：L3 的每个核心痛点都有功能解决方案？（对照 L3 痛点列表）
- [ ] 技术投入合理：无过度设计的功能？（每个功能都能追溯到场景）
- [ ] 功能复用性：至少有功能被多个场景复用？（不是 1:1 映射的标志）

## 框架检查

- [ ] 05-场景功能/CLAUDE.md 已生成？（约束摘要 + 强制回溯提醒 + 目录结构说明）
- [ ] 05-场景功能/CLAUDE.md 的约束摘要控制在 200 字以内？

- [ ] 项目 index.md 的 L5 进度标记已更新为 [x]？
- [ ] 项目根 CLAUDE.md 未被修改？（框架宪法，只读）

## 常见错误自查

- [ ] 是否避免了功能写成场景描述？（功能描述"系统怎么实现"，不是"用户做了什么"）
- [ ] 是否避免了 spec 重复定义领域知识？（规则在 models/rules/ 定义一次，spec 只引用）
- [ ] 是否避免了 API 设计脱离页面规格？（从 spec 的数据绑定和交互行为推导）
- [ ] 是否避免了状态机遗漏异常转换？（L4 异常场景的转换必须体现）
- [ ] 是否避免了产品上下文新增维度？（L5 只引用上层产出，不新增维度）
