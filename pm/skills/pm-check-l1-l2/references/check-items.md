# L1↔L2 跨层检查清单（11 项）

## 判定说明

跨层检查使用三级判定：
- **✅ 通过**：符合标准
- **⚠️ 有风险**：存在潜在问题，需要人判断是否可接受
- **❌ 不通过**：明确存在断裂或矛盾

## A. 约束传递（4 项）

### A1. 核心目的→业务事件
- **判定标准**：L1 的每个核心目的在 L2 主体链接图中都有对应的业务事件覆盖
- **检查方法**：
  1. 从 L1 index.md 提取核心目的列表
  2. 逐条在 L2 主体链接图中查找对应业务事件
  3. 记录缺失项
- **✅ 通过**：每个核心目的都有对应事件
- **❌ 不通过**：有核心目的在 L2 中无对应事件

### A2. 验收标准→纳入事件
- **判定标准**：L1 的验收标准在 L2 边界表中有对应的纳入事件
- **检查方法**：
  1. 从 L1 提取验收标准
  2. 在 L2 边界表中查找能验证该标准的纳入事件
- **✅ 通过**：验收标准可在 L2 业务模型中被验证
- **❌ 不通过**：有验收标准在 L2 中无法验证

### A3. 核心特征→边界体现
- **判定标准**：L1 的核心特征（含安全/底线）在 L2 边界表中有体现
- **检查方法**：
  1. 从 L1 提取核心特征
  2. 在 L2 边界表中检查是否有对应的纳入事件或边界规则
- **✅ 通过**：核心特征在 L2 中有体现
- **❌ 不通过**：有核心特征被 L2 忽略

### A4. 下游须知遵守
- **判定标准**：L1 CLAUDE.md 的下游须知在 L2 产出中被遵守
- **检查方法**：
  1. 从 L1 CLAUDE.md 读取下游须知
  2. 检查 L2 产出是否违反了须知中的约束
- **✅ 通过**：L2 遵守了所有下游须知
- **❌ 不通过**：L2 违反了 L1 的下游须知

## B. 一致性（4 项）

### B1. 上下文 L1 部分一致
- **判定标准**：L2 产品上下文的 L1 部分与 01-业务目的/index.md 一致
- **检查方法**：对比 L2 上下文中 L1 部分的关键信息与 L1 index.md
- **✅ 通过**：关键信息一致
- **❌ 不通过**：有信息被篡改或遗漏

### B2. 边界不矛盾
- **判定标准**：L2 的"不做什么"没有与 L1 核心目的矛盾
- **检查方法**：检查 L2 边界表中"不纳入"的事件是否与 L1 核心目的冲突
- **✅ 通过**：不纳入项与 L1 无矛盾
- **❌ 不通过**：L2 排除了 L1 核心目的要求的场景

### B3. 事件范围匹配
- **判定标准**：L2 主体链接图的事件范围与 L1 定义的业务范围匹配
- **检查方法**：对比 L1 的业务定义和 L2 的事件覆盖范围
- **✅ 通过**：范围一致
- **⚠️ 有风险**：L2 事件范围超出 L1 定义（可能是合理的业务深化）
- **❌ 不通过**：L2 事件范围明显偏离 L1 定义

### B4. 第一性原理不偏离
- **判定标准**：L1 的第一性原理在 L2 中没有被偏离
- **检查方法**：检查 L2 的整体设计是否仍在 L1 定义的业务本质范围内
- **✅ 通过**：L2 设计仍在第一性原理范围内
- **❌ 不通过**：L2 的设计方向偏离了 L1 的第一性原理

## C. 断裂检测（3 项）

### C1. 新概念合理性
- **判定标准**：L2 是否出现了 L1 未提及的新概念
- **检查方法**：
  1. 从 L2 中提取所有核心概念（主体、事件、实体）
  2. 检查哪些在 L1 中未提及
  3. 判断每个新概念是合理的业务发现还是偏离
- **✅ 通过**：无新概念，或新概念都是合理的业务发现
- **⚠️ 有风险**：有新概念，需要人判断合理性
- **❌ 不通过**：新概念偏离了 L1 定义的业务范围

### C2. 场景遗漏
- **判定标准**：L2 是否遗漏了 L1 明确要求的场景或需求
- **检查方法**：从 L1 的核心目的和验收标准中提取隐含场景，检查 L2 覆盖
- **✅ 通过**：无遗漏
- **❌ 不通过**：L1 要求的场景在 L2 中缺失

### C3. 验收可验证性
- **判定标准**：L1 的验收标准是否能在 L2 的业务模型中被验证
- **检查方法**：每个验收标准在 L2 中找到可验证的闭环
- **✅ 通过**：所有验收标准都有对应的业务闭环
- **❌ 不通过**：有验收标准在 L2 业务模型中无法验证
