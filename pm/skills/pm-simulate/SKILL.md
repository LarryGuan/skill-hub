# /pm-simulate：小白用户模拟

## 你的角色

你是可用性测试主持人，专门通过角色扮演 sub-agent 模拟真实用户走查产品，发现 PM 认知盲区中的体验问题。

核心验证维度：**"不知道系统设计的小白用户，能不能自然地完成目标？"**

你能发现四类问题：
- **迷惑🟡**：用户不理解为什么要这样操作，找不到入口
- **死路🔴**：用户想做的事系统不支持，无法继续
- **摩擦🟠**：能做但步骤太多、不直觉
- **错失🔵**：用户期待有但系统没有的功能

和 pm-test 的区别：
- pm-test：PM 自测，发现结构性问题（断链、缺口）
- pm-simulate：小白模拟，发现体验性问题（迷惑、死路、摩擦、错失）

## 执行流程

### 步骤 1：确认项目

从调用参数获取项目名。如果没有，询问用户。

检查 `Project/{项目名}/` 是否存在：
- **不存在** → 提醒用户先运行 pm-init
- **已存在** → 检查 `05-场景功能/` 目录是否有内容

检查 L5 是否已完成：
- 读取 `Project/{项目名}/index.md`，确认 L5 状态
- **L5 未完成** → 提醒用户先完成 L5（pm-simulate 依赖 L5 的页面规格）
- **L5 已完成** → 继续

### 步骤 2：加载上下文

读取三类核心文件：

1. **L3 画像**（构造角色卡的素材）
   - `Project/{项目名}/03-边界用户/index.md` → 用户画像总览
   - 提取每个画像的身份、技术水平、使用习惯、心理特征

2. **L4 场景**（构造任务目标的素材）
   - `Project/{项目名}/04-用户场景/index.md` → 场景总览
   - `Project/{项目名}/04-用户场景/{终端}/scenario-*.md` → 具体场景
   - 提取触发条件和用户目标

3. **L5 页面规格**（构造界面描述的素材）
   - `Project/{项目名}/05-场景功能/{终端}/pages/{页面名}/spec.md` → 页面规格
   - `Project/{项目名}/05-场景功能/{终端}/pages/{页面名}/wireframe.md` → 页面原型（如有）
   - 从每个 spec 提取用户可感知部分（布局、元素、反馈），移除技术术语

### 步骤 3：梳理测试矩阵

整理所有需要走查的组合：**用户 x 终端 x 场景**

输出测试矩阵，让用户确认：

```
测试矩阵：
| # | 用户角色 | 终端 | 场景 | 优先级 |
|---|---------|------|------|--------|
| 1 | 水果卖家 | 微信小程序 | 客户询价 | 高 |
| 2 | 水果买家 | 微信小程序 | 查看报价 | 高 |
| 3 | ... | ... | ... | ... |
```

优先级规则：
- 每个人类用户角色至少走查一次
- 每个终端至少走查一次
- 核心场景（L1 核心目的相关）优先

向用户确认矩阵后继续。

### 步骤 4：逐组合走查

对矩阵中的每个组合，执行三阶段流程：**走查说明 → 走查记录 → 走查报告**。

---

#### 阶段 A：构造走查说明（主 Agent）

##### 4a. 构造角色卡

从 L3 画像提取，过滤所有设计语言：
- 只保留用户能感知的描述（身份、技术水平、使用习惯、心理状态）
- 去掉"能力"、"需求"、"痛点"、"模块"等产品语言
- 读起来应该像在介绍一个你认识的人

##### 4b. 构造任务目标

从 L4 场景触发条件转化为日常语言：
- 目标必须具体到用户能理解的一句话
- 不能包含系统操作指引（如"使用批量修改功能"）
- 应该激发用户的自然行为

##### 4c. 准备界面描述

**V2 核心：从本终端所有页面的 spec 提取用户可感知部分。**

对每个页面：
- 提取页面名称、布局、可交互元素、状态变化、反馈信息
- 移除所有技术术语（后端逻辑、接口、状态机、权限判断内部逻辑）
- 用日常语言描述（"一个红色的大按钮"，而非"CTA 按钮"）

##### 4d. 构造 sub-agent prompt + 写入走查说明

按 methodology.md 中的 Sub-agent Prompt 模板组装：
- 角色代入指令
- 角色卡
- 设备环境
- 目标
- **本终端所有页面的描述**（V2 核心——不是单个页面）
- 行为指引 + 输出格式

**绝对不能包含**：系统设计意图、功能模块说明、数据库结构、API 接口、L1-L4 推导过程、page-flow。

将完整的走查说明写入：
`Project/{项目名}/06-验收测试/simulations/{终端}/sim-{角色}-{场景}-1走查说明.md`

走查说明包含：
1. **读取记录**——本次走了哪些 L3/L4/L5 文件（文件路径列表）
2. **原始素材摘要**——从原始文件中提取的关键信息
3. **术语过滤对照**——过滤了哪些产品术语，如何转化为日常语言
4. **角色卡**——最终版
5. **任务目标**——最终版
6. **sub-agent prompt 全文**——完整的、将发送给 sub-agent 的 prompt

##### 4d.1. 预准备走查记录文件

写入走查说明后，主 Agent 必须立即：

1. 创建目录：`mkdir -p Project/{项目名}/06-验收测试/simulations/{终端}/`
2. 写入 stub 文件到 `sim-{角色}-{场景}-2走查记录.md`，内容为元数据头：

```markdown
# 走查记录：{角色名} x {终端} x {场景名}

> 来源标记：待定
> 本文件由 sub-agent 在探索完成后写入。如果 sub-agent 未能写入，主 Agent 将从 tool result 回填。
```

---

#### 阶段 B：sub-agent 走查（Sub-agent）

##### 4e. Agent 调用 sub-agent

使用 Agent tool 调用 sub-agent，每个场景一次独立调用。Sub-agent 拿到本终端所有页面，自由探索。

Sub-agent prompt 中必须包含走查记录的**确切文件路径**：
`Project/{项目名}/06-验收测试/simulations/{终端}/sim-{角色}-{场景}-2走查记录.md`

**Sub-agent 的写入要求**：
- 探索完成后，sub-agent 使用 Write tool 将完整探索记录写入上述指定文件路径
- 记录思考过程：怎么思考的、什么目标、想做什么、预期什么感受
- 记录探索过程：每步怎么走的、看到什么、做了什么
- 记录每步感受和问题
- Sub-agent 自由选择格式，保持原始性

走查记录的内容包括但不限于：
- 逐步探索过程（我看到 / 我想做 / 我操作了 / 结果 / 我的感受）
- 遇到的困惑、死路、摩擦、错失
- 做得好的地方

##### 4e.1. 写入后检查

主 Agent 收到 Agent tool 返回后，检查 `2走查记录.md` 是否被 sub-agent 成功写入：

1. 从磁盘 Read 该文件
2. 判断内容：
   - **内容超过 stub header** → sub-agent 成功写入，正常继续
   - **内容仅为 stub header** → sub-agent 未写入，触发回填：
     1. 从 Agent tool 返回值中提取 sub-agent 的完整输出
     2. 将完整输出写入 `2走查记录.md`（覆盖 stub）
     3. 将来源标记改为"主Agent从tool result回填"

---

#### 阶段 C：编译走查报告（主 Agent）

##### 4f. 读取走查说明 + 走查记录

**主 Agent 始终从磁盘读取 `2走查记录.md`（不依赖 Agent tool 返回值或上下文记忆）。**

主 Agent 从磁盘读取：
- `sim-{角色}-{场景}-1走查说明.md`——回溯构造过程
- `sim-{角色}-{场景}-2走查记录.md`——sub-agent 的原始探索

##### 4g. 编写走查报告

按 output-template.md 格式编译：

**如果是回填的走查记录，在走查报告中标注来源。**

1. **整理为标准格式**——把 sub-agent 的自由格式记录转化为结构化报告
2. **质量评估**——按 quality-checklist.md 5 大类逐项检查，评价 sub-agent 走查质量
3. **问题归类**——按四种类型（迷惑🟡/死路🔴/摩擦🟠/错失🔵）整理问题清单
4. **总结评定**——走查顺利度评分、完成目标情况、总耗时感

写入 `Project/{项目名}/06-验收测试/simulations/{终端}/sim-{角色}-{场景}-3走查报告.md`

写入前，按 quality-checklist.md 逐项检查。任何不通过项必须补全后再写入。

### 步骤 5：汇总所有走查结果

所有组合走查完成后，输出汇总报告：

```
模拟走查汇总：
- 总走查数：X 次
- 发现问题总数：Y 个
  - 迷惑🟡：a 个
  - 死路🔴：b 个（高优先级）
  - 摩擦🟠：c 个
  - 错失🔵：d 个

高优先级问题（死路）：
1. [描述] — 涉及 spec-xxx — 建议：...
2. ...

需要特别关注的设计盲区：
- [汇总各角色的共性问题]

各走查质量评估：
- [各场景的走查质量评级]
```

## 注意事项

### 三文件机制
- **走查说明**（1走查说明.md）是主 Agent 的留底——记录从原始文件到 sub-agent prompt 的完整推导过程
- **走查记录**（2走查记录.md）是 sub-agent 的原始输出——主 Agent 不修改，保持保真
- **走查报告**（3走查报告.md）是主 Agent 的分析——结构化整理 + 质量评估 + 问题总结
- 三个文件共同构成一次走查的完整证据链

### Sub-agent 使用说明
- 使用 Agent tool 调用 sub-agent，每个场景组合一次独立调用
- Sub-agent 之间**绝对不能共享上下文**——每个 sub-agent 是一个全新的、不知道系统设计的新用户
- 给 sub-agent **本终端所有页面**，不是单个页面。这是 V2 的核心变化——sub-agent 自由探索，能发现 page-flow 之外的盲区
- Sub-agent 探索完成后用 Write tool 写入指定文件路径。主 Agent 写入后检查，未写入则从 tool result 回填

### 走查记录写入的可靠性保障

走查记录是走查报告的核心输入，必须确保写入成功。采用三层保障机制：

1. **预准备**（主 Agent 负责）：在调用 sub-agent 之前，主 Agent 创建目标目录并写入 stub 文件，确保文件路径存在、目录权限正确
2. **Prompt 强化**：sub-agent prompt 中明确指定文件路径，并将"使用 Write tool 写入完整探索记录"作为独立、强制的最后一步
3. **检查 + 回填**（主 Agent 负责）：收到 Agent tool 返回后，主 Agent 从磁盘读取文件，检查是否超过 stub header。未写入则从 tool result 提取完整输出回填

### 角色卡隔离要求
- 角色卡中**不能出现任何产品术语**（"模块"、"API"、"实体"、"功能"等）
- Sub-agent prompt 中**不能包含设计意图**——sub-agent 不知道系统为什么这样设计
- Sub-agent prompt 中**不能包含 page-flow**——让用户自己决定怎么走

### 质量底线
- 每个问题必须有修改建议，不能只说"这里不好"
- 必须记录做得好的地方，不能只有问题
- 问题按严重程度分级：高（死路）> 中（迷惑）> 低（摩擦）
- 走查报告必须评价 sub-agent 的走查质量（走查是否充分、是否真的代入角色）
