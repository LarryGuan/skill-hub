# 模拟走查方法论

## 核心原则：角色卡隔离

Sub-agent 只知道三件事：
1. **我是谁**（身份、技术水平、心理状态）
2. **我在哪**（设备、环境、网络）
3. **我要干嘛**（目标，用日常语言描述）

Sub-agent **不知道**：
- 系统为什么这样设计
- 有哪些功能模块
- 后端数据结构
- 设计决策的权衡过程
- L1-L4 的推导过程
- page-flow（页面之间的设计跳转逻辑）

---

## 角色卡构造方法

### 从 L3 提取画像

L3 的用户画像通常包含设计分析语言，需要过滤：

| L3 原文 | 过滤后（角色卡） |
|---------|-----------------|
| "作为核心使用者，需要产品管理能力" | "每天要管上百个产品的价格" |
| "技术熟练度：中级" | "微信用得溜，但不太会用电脑" |
| "使用场景：市场摊位、碎片时间" | "在市场摊位上用手机，一会儿一个客户" |
| "核心痛点：手写报价太慢" | "以前手写报价单，一个要十几分钟，客户等不及" |

### 过滤原则
- 只保留用户能感知到的描述
- 去掉所有"能力"、"需求"、"痛点"、"模块"、"功能"等产品语言
- 保留年龄、职业、技术水平、使用习惯等鲜活信息
- 角色卡读起来应该像在介绍一个你认识的人

---

## 任务目标构造

从 L4 场景提取触发条件，转化为用户的自然语言目标。

### 转化规则

| L4 场景描述 | 转化为用户目标 |
|------------|---------------|
| "卖家在市场接到客户询价" | "你正在市场摊位上，老客户王老板微信问你'红富士什么价'" |
| "买家通过分享链接查看产品册" | "你收到一个卖家的微信消息，里面有他产品的报价单" |
| "卖家需要批量修改产品价格" | "最近苹果涨价了，你需要把所有苹果产品的价格调高 10%" |

### 关键要求
- 目标必须具体到用户能理解的一句话
- 目标不能包含系统操作指引（如"使用批量修改功能"）
- 目标应该激发用户的自然行为，而非引导用户走特定路径

## 界面描述准备（V2 核心：所有页面）

### V2 关键变化

V1 只给 sub-agent 单个页面逐步走查。V2 给 sub-agent **本终端所有页面**，让 sub-agent 自由探索。

这意味着：
- Sub-agent 自己决定先看哪个页面、按什么顺序探索
- Sub-agent 可能走出 PM 没设计的路径（这是特性，不是 bug）
- 能发现 page-flow 之外的盲区——比如用户想找"设置"但没有设置页

### 提取方法

从 `05-场景功能/{终端}/pages/{页面名}/spec.md` 提取每个页面的用户可感知部分：

1. **页面名称**：用户看到的标题/名称
2. **页面布局**：从上到下、从左到右，用户能看到什么区域
3. **可交互元素**：按钮、输入框、链接、列表项——用日常语言描述（"一个红色的大按钮"，而非"CTA 按钮"）
4. **页面状态**：空状态、加载中、有数据时——用户会看到什么不同
5. **反馈信息**：操作后的提示、弹窗、跳转

### 过滤规则

| Spec 中的内容 | 处理方式 |
|--------------|---------|
| 布局结构 | 保留，翻译为用户可感知的描述 |
| 交互元素 | 保留，移除技术术语 |
| 数据规则 | 只保留用户可见的（如"最多选 20 个"） |
| 后端逻辑 | 移除 |
| 接口调用 | 移除 |
| 状态机 | 移除 |
| 权限判断 | 只保留用户可感知的结果（如"灰色不可点"） |
| wireframe 参考 | 保留布局感知，翻译为文字描述 |

## 问题分类法

### 四种问题类型

1. **迷惑** 🟡
   - 用户不理解为什么要这样操作
   - 找不到操作入口
   - 不确定操作的结果
   - 严重程度：中

2. **死路** 🔴
   - 用户想做的事但系统不支持
   - 操作到一半卡住了，无法继续
   - 数据丢失或状态不一致
   - 严重程度：高

3. **摩擦** 🟠
   - 能做但步骤太多
   - 需要反复思考"下一步该点哪"
   - 操作不符合直觉
   - 严重程度：低~中

4. **错失** 🔵
   - 用户期待有但系统没有的功能
   - 用户尝试做某事但找不到入口
   - 严重程度：视情况

## Sub-agent Prompt 构造

### V2 核心机制：自由探索

每个场景组合（用户 x 终端 x 场景）= 一次 Agent 调用。Sub-agent 拿到角色卡 + 目标 + **本终端所有页面**，自由探索。

为什么要给所有页面？
1. 发现 page-flow 之外的盲区——小白想找"设置"但没有设置页
2. 用户自由选择探索路径——不限于 PM 设计的 page-flow 顺序
3. 实现简单可靠——每个场景一次原子调用，不需要多轮编排
4. 角色卡保证视角——虽然看到所有页面，但小白角色决定他关注什么

### 必须包含的部分

1. **角色代入指令**："你是一个真实的人，不是 AI 助手"
2. **角色卡**：身份 + 技术水平 + 环境 + 心理状态
3. **设备环境**：终端 + 网络条件 + 物理环境
4. **目标**：一句话，用日常语言
5. **所有页面描述**：本终端所有页面的用户可感知描述（不是单个页面）
6. **行为指引**：自由探索，按直觉行动，找不到就说找不到
7. **文件写入指令**：探索完成后用 Write tool 写入指定路径（独立、强制的最后一步）

### 绝对不能包含的部分

- 系统设计意图
- 功能模块说明
- 数据库结构
- API 接口说明
- L1-L4 的推导过程
- page-flow（页面跳转设计）
- pm-test 的测试用例

### Sub-agent Prompt 模板

```
你是一个真实的人，不是 AI 助手。请完全代入以下角色：

【你是谁】
[角色卡内容，日常语言]

【你的设备】
[设备环境描述]

【你的目标】
[任务目标，日常语言]

【你可以探索的产品页面】
[本终端所有页面的用户可感知描述，包含布局和可交互元素]

请开始探索这个产品。你可以自由地浏览任何你感兴趣的页面，按你的直觉行动。

如果你想找某个功能但找不到，直接说"找不到"。
遇到困惑就直说，不要猜系统的意图。你是一个从没用过这个系统的新用户。

---

在下方记录你的探索过程。有什么写什么，不要求格式化：
- 你的思考：在想什么、想做什么、预期什么感受
- 你的探索：每步怎么走的、看到什么、做了什么
- 你的感受：每步的感觉、有没有问题、什么问题

探索完成后，给出你的整体评价。

---

⚠️ 完成后必须执行（最重要的一步）：

探索和评价全部写完后，你必须使用 Write tool，将上面写的全部内容写入以下文件：

Project/{项目名}/06-验收测试/simulations/{终端}/sim-{角色}-{场景}-2走查记录.md

用 Write tool 将完整探索记录写入上述路径。覆盖已有内容。
这是你必须完成的最后一步，不能省略。
```

## 走查记录写入可靠性

### 三层保障机制

**第一层：主 Agent 预准备**
- 主 Agent 在调用 sub-agent 前，创建目标目录（`mkdir -p`）
- 主 Agent 写入 stub 文件到 2走查记录.md 路径（含 metadata header）
- 消除"目录不存在"的写入失败风险

**第二层：Prompt 强化**
- 文件写入指令是 prompt 中的独立段落，用 `⚠️` 标记醒目
- 明确指定完整文件路径
- 定位为"探索完成后必须执行的最后一个动作，不能省略"

**第三层：检查 + 回填**
- 主 Agent 收到 tool result 后，读取 2走查记录.md 检查是否被 sub-agent 写入
- 已写入（内容超过 stub header）→ 正常继续
- 未写入 → 主 Agent 从 tool result 提取 sub-agent 完整输出回填到文件
- 回填时在文件头标注来源为"主Agent从tool result回填"

## 常见误区

1. **角色卡包含设计知识** → Sub-agent 就不是"小白"了，会做出设计者期望的操作而非真实用户的操作
2. **用产品术语描述界面** → "CTA 按钮"不是用户语言，"一个红色的大按钮"才是
3. **只给单个页面** → V1 的做法，V2 必须给本终端所有页面，让 sub-agent 自由探索
4. **只记录问题不记录好的地方** → 好的体验同样有价值，帮助确认设计决策
5. **所有问题都标高严重程度** → 失去区分度，真正的死路问题被淹没
6. **Sub-agent 之间共享上下文** → 每个 sub-agent 必须是独立调用（独立 Agent 调用），不继承其他角色的记忆
7. **引导 sub-agent 走特定路径** → 不要在 prompt 中暗示"先看首页再点XX"，让用户自己决定
8. **修改走查记录内容** → 走查记录（2走查记录.md）保持保真，即使是回填也不修改 sub-agent 的原始输出
9. **走查说明不留底** → 必须写入 1走查说明.md，记录从原始文件到 sub-agent prompt 的完整推导过程
10. **走查报告缺少质量评估** → 必须评价 sub-agent 的走查质量（是否充分、是否代入角色）
11. **忽略写入后检查** → 主 Agent 必须检查 sub-agent 是否写入了文件，未写入则触发回填
