# L2 方法论精华

## 核心方法：主体链接图

主体链接图是 L2 的核心产出，一张图统领所有其他产出。

### 绘制规范

```
graph TD
    买家 -->|"1.浏览商品[小程序]"| 平台
    买家 -->|"2.搜索比价[PC网页]"| 平台
    买家 -->|"3.下单支付[小程序/PC网页]"| 平台
    卖家 -->|"4.接单处理[PC网页/App]"| 平台
    平台 -->|"5.通知发货[App]"| 物流
    物流 -->|"6.配送更新[小程序]"| 买家
```

**要素**：
- **节点** = 主体（粗粒度，如"买家""卖家""平台""物流"）
- **连线** = 有向事件（A 对 B 做了什么）
- **线上标注** = 序号 + 事件描述 + 终端归属 `[终端类型]`
- **格式** = Mermaid graph

**绘制顺序**：从业务场景的起点开始，按事情发展推进标序号。序号代表事务发生的过程。

**终端标注规则**：
- 同一事件可能跨终端，用 `/` 分隔：`[小程序/PC网页]`
- 如果不确定终端归属，先不标注，在阶段二确认终端后再补

### 主体识别方法

从 L1 核心目的和用户原始想法中提取主体。

**识别维度**：
1. **直接参与者**：谁在使用这个系统？（最显而易见的角色）
2. **间接参与者**：谁在流程中被影响但不是主动使用者？（如物流、财务）
3. **系统角色**：平台、管理员等系统性角色

**粒度控制**：
- 主体应该是粗粒度的角色，不是具体的人
- ✅ "买家""卖家""平台""物流"
- ❌ "张三""李四""客服小王"

**引导问题**：
- "还有谁参与？"（扩大范围）
- "这件事谁发起？谁响应？谁被动参与？"（明确方向）
- "流程走完，涉及哪些角色？"（查漏补缺）

### 链接图推导法

从主体链接图推导其他产出物的方法：

**推导类图**：
- 主体 → 实体（每个主体可能是一个实体）
- 事件 → 关系/操作（实体之间的交互映射为关联关系）
- 关注实体的属性骨架（不需要完整字段，只需核心属性）

**推导用例图**：
- 纳入系统的事件 → 用例
- 主体 → 参与者（Actor）
- 系统边界由"做什么/不做什么"决定

## 核心方法：终端确认

### 终端类型识别

引导用户确认每个主体使用的终端类型。

**常见终端类型**：
- 小程序（微信/支付宝等）
- App（iOS/Android）
- H5（移动端浏览器）
- PC网页
- POS机/硬件终端
- API/开放平台

**引导问题**：
- "这个主体在什么设备上操作？"
- "需要移动端还是PC端？"
- "有没有硬件终端参与？"

### 终端场景清单提取

从主体链接图按终端分组，提取每个终端的事件序列。

**提取方法**：
1. 遍历主体链接图的所有事件
2. 按终端归属分组（跨终端的事件在多个终端都出现）
3. 在每个终端内按序号排列
4. 标注涉及主体和定位

**示例**：

| 终端 | 事件序列 | 涉及主体 | 定位 |
|------|---------|---------|------|
| 小程序 | 1.浏览商品 → 3.下单支付 → 6.查看物流 | 买家 | 核心购物流程 |
| PC网页 | 2.搜索比价 → 4.接单处理 → 5.批量采购 | 买家, 卖家 | 管理端偏重 |
| App | 4.接单处理 → 5.通知发货 | 卖家, 物流 | 运营端 |

**定位判断**：
- 核心流程：终端上串联了完整的用户关键路径
- 管理端：偏向查询、管理、配置
- 运营端：偏向业务执行、状态更新

### 终端分工表生成

从终端确认结果生成终端分工表。

**表格结构**：

| 主体 | 终端类型 | 能力边界 |
|------|---------|---------|
| 买家 | 小程序 | 浏览、购买、追踪 |
| 买家 | PC网页 | 搜索比价、批量采购 |
| 卖家 | PC网页 | 商品管理、订单处理 |
| 卖家 | App | 订单通知、紧急处理 |
| 物流 | App | 配送更新、路线查看 |

**能力边界**：该终端上这个主体能做的事情范围，不是功能列表，是业务能力描述。

## 核心方法：边界判断

### 做什么 / 不做什么

从主体链接图的事件中划分系统边界。

**纳入判定标准**：
- 事件需要系统支持（存储、计算、通知等）
- 事件是核心业务流程的一部分
- 事件涉及数据采集或状态变更

**不纳入判定标准**：
- 事件完全在线下完成，系统无需介入
- 事件属于第三方系统负责（如支付由支付平台处理，系统只对接结果）
- 事件在当前阶段不做（MVP 不考虑）

**每条判定的格式**：

| 事件 | 判定 | 原因 |
|------|------|------|
| 买家浏览商品 | 纳入 | 核心业务流程，需要商品数据展示 |
| 买家线下看样 | 不纳入 | 完全线下行为，系统无法介入 |
| 支付平台扣款 | 不纳入 | 第三方系统负责，系统只接收支付结果 |

### 边界与终端的关系

边界划分需考虑终端约束：
- 某事件在特定终端上是否可行？（如复杂报表在手机上不现实）
- 某事件在特定终端上是否有必要？（如简单的状态查看不需要 PC）
- 跨终端的事件，是否两端都纳入？

## 推导方法

### 类图初版推导

从主体链接图推导类图初版的步骤：

1. **识别实体**：主体 → 实体（不一定一一对应，一个主体可能拆成多个实体）
2. **识别关系**：事件 → 实体间的关联（一对多、多对多等）
3. **识别属性骨架**：从事件中推断实体需要哪些核心属性
4. **识别操作**：从事件中推断实体的核心方法

**粒度控制**：
- 类图初版只需实体名 + 核心属性 + 关系
- 不需要完整字段类型和访问修饰符
- L5 会在此基础上细化

**格式**：Mermaid classDiagram

### 用例图初版推导

从事件和系统边界推导用例图的步骤：

1. **筛选纳入事件**：只取边界表中"纳入"的事件
2. **映射用例**：每个纳入事件 → 一个用例
3. **映射参与者**：事件的发起主体 → Actor
4. **标注终端约束**：用例发生在哪个终端

**粒度控制**：
- 用例图初版只画核心用例，不画包含/扩展关系
- L4 会在此基础上细化为场景级用例

**格式**：Mermaid 用例图（可用 graph 模拟，节点用例用圆角矩形）

## 常见误区

1. **主体太细**：把具体的人当成主体 → 应该用角色名，不要用人名
2. **事件包含实现细节**："点击提交按钮" → 应该是"提交订单"
3. **遗漏间接主体**：只看到直接用户，忘了物流、财务等间接参与者
4. **边界太宽**：什么都想做 → 明确 MVP 范围，区分"现在做"和"以后做"
5. **终端遗漏**：只想到主终端，忘了次要终端或硬件终端
6. **跨终端事件不标注**：同一事件在多个终端都能做，但没有标注 → 导致 L4 缺场景
7. **终端场景清单与图不一致**：清单里的事件序号和图上对不上

## 追问技巧

- 用户只说"用户" → 追问："是哪种用户？买家还是卖家？他们做的事情一样吗？"
- 用户直接说功能 → 追问："这是哪个角色在什么场景下要做的？"
- 用户说"所有人" → 追问："平台管理员也是吗？他们和普通用户的区别是什么？"
- 用户卡住 → 提供同行业的主体链接图示例，但标注"这是参考示例"
- 终端不确定 → 追问："你平时用什么设备做这件事？你的客户呢？"
