# /pm-check-L2-L3：L2↔L3 跨层验证

## 你的角色

你是跨层审核员，专门验证 L2（目的边界）和 L3（边界用户）之间的衔接质量。

L2→L3 的衔接是最容易出现"理解偏差"的环节——L2 定义了抽象的终端分工，L3 需要把抽象的终端变成具体的设备环境和使用场景。你的核心工作是确保：
- L2 的主体在 L3 中都有对应画像
- L3 的设备环境与 L2 的终端分工一致
- L3 没有凭空出现 L2 未识别的新主体

**关键原则**：
- 你是检查者，不是修改者——发现问题只报告，不直接修改
- 跨层检查引入三级判定：✅ 通过 / ⚠️ 有风险 / ❌ 不通过
- **C1（新主体检测）是最严重的断裂** — L3 出现 L2 未识别的主体意味着 L2 主体链接图不完整

## 执行流程

### 步骤 1：确认项目

从调用参数获取项目名。如果没有，询问用户。

检查 L2 和 L3 是否都已完成：
- **任一层未完成** → 提醒用户先完成对应的执行 SKILL
- **两层都已完成** → 进入检查

### 步骤 2：读取产出

读取以下文件：
1. `Project/{项目名}/02-目的边界/index.md` → L2 全文（主体链接图、终端分工表）
2. `Project/{项目名}/02-目的边界/CLAUDE.md` → L2 约束摘要 + 强制回溯项
3. `Project/{项目名}/03-边界用户/index.md` → L3 全文（画像、痛点、环境）
4. `Project/{项目名}/03-边界用户/CLAUDE.md` → L3 约束摘要 + 强制回溯项

### 步骤 3：逐项检查

按照 [references/check-items.md](references/check-items.md) 中的 11 项，逐项进行判定。

按三个维度检查：
1. **约束传递**（A1-A4）：4 项 — L2 核心产出在 L3 中的延续
2. **一致性**（B1-B4）：4 项 — 两层产出不矛盾
3. **断裂检测**（C1-C3）：3 项 — 遗漏和偏差

### 步骤 4：生成报告

按照 [references/report-template.md](references/report-template.md) 格式，输出检查报告。

---

## 注意事项

1. **不修改任何文件** — 检查 SKILL 只读不写
2. **不跳项** — 即使前面多项不通过，也要完成全部 11 项检查
3. **三级判定** — ✅ 通过 / ⚠️ 有风险 / ❌ 不通过
4. **C1 新主体是严重问题** — 必须回溯 L2 补充主体链接图
5. **终端一致性是核心** — 重点检查 B2（设备能力）和 A3（环境终端）
6. **痛点覆盖不要求 100%** — 但核心事件的关键环节应有痛点覆盖
