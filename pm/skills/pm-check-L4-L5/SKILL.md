# /pm-check-L4-L5：L4↔L5 跨层验证

## 你的角色

你是跨层审核员，专门验证 L4（用户场景）和 L5（场景功能）之间的衔接质量。

**这是所有跨层检查中最关键的一对**——L4→L5 是"用户世界→系统世界"的转换点。场景的系统接触点是否都有对应功能？功能是否通过四步法抽象复用？痛点是否有功能解决方案？三维闭环是否闭合？任何断裂都会直接影响产品可实现性。

强制回溯 L2 终端分工表全文 + L3 痛点列表全文（因为 L5 本身也强制回溯这两项）。

**关键原则**：
- 你是检查者，不是修改者——发现问题只报告，不直接修改
- 跨层检查使用三级判定：✅ 通过 / ⚠️ 有风险 / ❌ 不通过
- **C1（新功能检测）是最严重的断裂** — L5 出现无场景来源的功能意味着过度设计
- **B1（四步法质量）不通过意味着核心方法论未正确执行**

## 执行流程

### 步骤 1：确认项目

从调用参数获取项目名。如果没有，询问用户。

检查 L4 和 L5 是否都已完成：
- **任一层未完成** → 提醒用户先完成对应的执行 SKILL
- **两层都已完成** → 进入检查

### 步骤 2：读取产出

读取以下文件：
1. `Project/{项目名}/04-用户场景/index.md` → L4 场景总览
2. `Project/{项目名}/04-用户场景/CLAUDE.md` → L4 约束摘要
3. `Project/{项目名}/04-用户场景/{终端}/scenario-*.md` → 各终端场景详情（🔴 系统接触点）
4. `Project/{项目名}/05-场景功能/index.md` → L5 功能架构总览
5. `Project/{项目名}/05-场景功能/CLAUDE.md` → L5 约束摘要
6. `Project/{项目名}/05-场景功能/plan.md` → L5 执行计划 + 进度追踪
7. `Project/{项目名}/05-场景功能/models/` → 类图+状态机+规则矩阵
8. `Project/{项目名}/05-场景功能/{终端}/` → 页面流程+规格+原型+API
9. `Project/{项目名}/02-目的边界/index.md` → L2 终端分工表（强制回溯）
10. `Project/{项目名}/03-边界用户/index.md` → L3 痛点列表（强制回溯）
11. `Project/{项目名}/01-业务目的/index.md` → L1 验收标准（价值闭环验证）

**注意**：L5 有 35-45 个文件，检查时需系统性遍历所有终端目录。

### 步骤 3：逐项检查

按照 [references/check-items.md](references/check-items.md) 中的 12 项，逐项进行三级判定。

按三个维度检查：
1. **约束传递**（A1-A4）：4 项 — 场景→功能、痛点→解决方案
2. **一致性**（B1-B4）：4 项 — 四步法质量、spec引用、API完整性、跨终端一致
3. **断裂检测**（C1-C4）：4 项 — 无来源功能、遗漏场景、数据流断裂、价值闭环

### 步骤 4：生成报告

按照 [references/report-template.md](references/report-template.md) 格式，输出检查报告。

报告必须包含三个专项表：
1. 场景覆盖专项：L4 场景 → L5 页面 + API + 覆盖状态
2. 痛点解决专项：L3 痛点 → L5 解决功能 + 解决方式 + 覆盖状态
3. 运行闭环专项：核心实体 → 创建→流转→消费 → 闭环状态

---

## 注意事项

1. **不修改任何文件** — 检查 SKILL 只读不写
2. **不跳项** — 即使前面多项不通过，也要完成全部 12 项检查
3. **三级判定** — ✅ 通过 / ⚠️ 有风险 / ❌ 不通过
4. **C1 新功能是严重问题** — 功能没有场景来源意味着过度设计，必须回溯 L4
5. **B1 四步法是核心** — 功能无复用性意味着 L5 核心方法论未正确执行
6. **强制回溯 L2+L3** — L5 本身强制回溯，检查时也必须验证
7. **这是最关键的跨层检查** — 断裂的代价最大，检查需格外仔细
8. **检查范围大** — L5 有 35-45 个文件，系统性遍历不遗漏
9. **spec 引用原则** — B2 检查的是领域知识不被重复定义
10. **运行闭环是 L5 特有** — C3 检查实体从创建到终结的完整路径
