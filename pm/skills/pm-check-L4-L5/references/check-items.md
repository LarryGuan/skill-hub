# L4↔L5 跨层检查清单（12 项）

## 判定说明

跨层检查使用三级判定：
- **✅ 通过**：符合标准
- **⚠️ 有风险**：存在潜在问题，需要人判断是否可接受
- **❌ 不通过**：明确存在断裂或矛盾

## A. 约束传递（4 项）

### A1. 场景→功能覆盖
- **判定标准**：L4 每个 🔴 系统接触点在 L5 功能中有对应（非 1:1 但全覆盖）
- **检查方法**：
  1. 从 L4 各终端场景文件中提取所有 🔴 系统接触点
  2. 从 L5 页面流程和功能规格中提取所有功能点
  3. 逐个比对系统接触点是否有功能覆盖
- **✅ 通过**：所有系统接触点都有功能覆盖
- **❌ 不通过**：有系统接触点未被任何功能覆盖

### A2. 痛点→功能解决方案
- **判定标准**：L3 每个核心痛点在 L5 功能中有明确的解决方案
- **检查方法**：
  1. 从 L3 痛点列表提取所有核心痛点
  2. 从 L5 功能规格和 API 中查找对应的解决功能
  3. 验证解决方案是否直接针对痛点描述的问题
- **✅ 通过**：所有核心痛点有功能解决方案
- **❌ 不通过**：有核心痛点无功能解决方案

### A3. 终端→页面终端一致
- **判定标准**：L4 场景的终端与 L5 页面流程/规格的终端一致
- **检查方法**：
  1. 从 L4 场景目录提取终端列表
  2. 从 L5 页面流程目录提取终端列表
  3. 逐个比对终端名称和数量
- **✅ 通过**：终端列表完全一致
- **❌ 不通过**：终端列表不一致

### A4. 强制回溯执行
- **判定标准**：L5 spec 正确引用了 models/ 的实体 ID、规则 ID、状态机 ID
- **检查方法**：
  1. 从 L5 models/ 提取所有实体 ID、规则 ID、状态机 ID
  2. 检查 L5 各终端 spec 文件中的引用是否指向上述 ID
  3. 检查是否有 spec 重复定义了 models/ 中已有的领域知识
- **✅ 通过**：spec 正确引用了 models/ 的产出
- **❌ 不通过**：spec 重复定义了领域知识，或引用不存在的 ID

## B. 一致性（4 项）

### B1. 功能复用性（非 1:1）
- **判定标准**：L5 功能通过四步法抽象，至少有功能被多个场景复用
- **检查方法**：
  1. 从 L5 功能架构中统计每个功能服务的场景数
  2. 计算复用率（被多个场景复用的功能 / 总功能数）
  3. 验证四步法是否被正确执行（场景识别→能力提取→通用化→组合复用）
- **✅ 通过**：有明确的多场景复用功能，复用率合理
- **⚠️ 有风险**：复用率较低，大部分功能只服务单个场景
- **❌ 不通过**：功能完全 1:1 映射场景，四步法未执行

### B2. spec 引用正确性
- **判定标准**：spec 引用了 class.md 实体字段、states/ 状态机、rules/ 规则，而非重复定义
- **检查方法**：
  1. 检查每个终端 spec 文件是否引用 models/ 中的实体和规则
  2. 检查是否有 spec 自行定义了字段、状态或规则（应引用而非重新定义）
  3. 统计重复定义的数量和严重程度
- **✅ 通过**：spec 只引用不重复定义
- **⚠️ 有风险**：个别 spec 有少量重复定义
- **❌ 不通过**：大量 spec 重复定义领域知识

### B3. API 推导完整性
- **判定标准**：每个页面的数据展示+提交需求都有对应 API 接口
- **检查方法**：
  1. 从每个页面的规格中提取数据展示需求和提交需求
  2. 从 API 文档中查找对应的接口
  3. 逐个验证需求是否有 API 覆盖
- **✅ 通过**：所有页面需求有 API 覆盖
- **⚠️ 有风险**：个别页面缺少非关键 API
- **❌ 不通过**：大量页面缺少 API

### B4. 跨终端数据一致
- **判定标准**：同一实体在不同终端 API 中的字段定义不矛盾
- **检查方法**：
  1. 提取每个实体在各终端 API 中的字段定义
  2. 对比同一实体在不同终端的字段名称、类型、约束
  3. 检查是否有矛盾（如同名字段不同类型）
- **✅ 通过**：跨终端字段定义一致
- **❌ 不通过**：有实体在不同终端的字段定义矛盾

## C. 断裂检测（4 项）

### C1. 新功能检测
- **判定标准**：L5 功能是否出现无 L4 场景来源的新功能
- **检查方法**：
  1. 从 L5 功能架构中提取所有功能列表
  2. 逐个功能回溯其在 L4 场景中的来源（哪个场景的系统接触点）
  3. 标记无法回溯到 L4 场景的功能
- **✅ 通过**：所有功能都有场景来源
- **❌ 不通过**：有功能无场景来源 → 必须回溯 L4 补充场景或删除该功能（最严重的断裂）

### C2. 遗漏场景
- **判定标准**：L4 场景是否有未被 L5 页面/API 覆盖的
- **检查方法**：
  1. 从 L4 场景列表提取所有场景
  2. 检查每个场景在 L5 页面流程或 API 中是否有对应
  3. 区分核心场景和非核心场景
- **✅ 通过**：所有 L4 场景被覆盖
- **⚠️ 有风险**：个别非核心场景未覆盖（有排除说明）
- **❌ 不通过**：核心场景未覆盖

### C3. 数据流断裂
- **判定标准**：跨终端的数据流是否闭合（实体创建→流转→消费路径完整）
- **检查方法**：
  1. 从 L5 models/ 提取核心实体列表
  2. 追踪每个实体在各终端 API 中的创建、读取、更新、删除路径
  3. 验证实体从创建到终结的完整生命周期是否闭合
- **✅ 通过**：所有核心实体的数据流闭合
- **⚠️ 有风险**：个别非核心实体数据流未闭合
- **❌ 不通过**：核心实体数据流断裂

### C4. 价值闭环缺失
- **判定标准**：L1 验收标准是否有 L5 功能支撑，无过度设计功能
- **检查方法**：
  1. 从 L1 验收标准提取所有价值承诺
  2. 从 L5 功能中查找支撑每个价值承诺的功能
  3. 反向检查是否有功能无法关联到任何价值承诺
- **✅ 通过**：L1 验收标准有功能支撑，无过度设计
- **⚠️ 有风险**：有少量可能过度设计的功能
- **❌ 不通过**：L1 验收标准无功能支撑，或大量过度设计
