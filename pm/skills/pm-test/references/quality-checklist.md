# 测试用例质量检查清单

## 完整性检查

- [ ] 每个用户×终端组合是否至少有一个测试用例？
- [ ] 每个测试用例是否有前置条件？
- [ ] 前置条件是否具体到数据状态？（不能只写"系统正常"）
- [ ] 覆盖率矩阵是否覆盖了所有 L5 spec？（遍历 `05-场景功能/{终端}/pages/{页面名}/spec.md`）
- [ ] 覆盖率矩阵是否暴露了未覆盖的 spec？
- [ ] 所有场景（来自 L4）是否都有对应的测试用例？

## 链式验证检查

- [ ] 每个测试用例的步骤是否链式衔接？（上一步输出 = 下一步输入）
- [ ] 链式衔接是否对照了 page-flow.md（`05-场景功能/{终端}/page-flow.md`）的页面流转？
- [ ] 断链是否已记录在"断链记录"中？
- [ ] 每个断链是否有影响分析？
- [ ] 步骤中的状态流转是否与 models/ 中的状态机一致？

## Spec 引用检查

- [ ] 每个步骤是否标注了对应的 spec？
- [ ] 引用的 spec 文件是否存在？（路径格式：`{页面名}/spec-xxx`，对应 `05-场景功能/{终端}/pages/{页面名}/spec.md`）
- [ ] 有没有步骤没有对应 spec？（说明功能设计有遗漏）
- [ ] 有没有 spec 被多个测试用例覆盖？（允许，但需确认是否有冲突）
- [ ] spec 引用是否覆盖了 wireframe.md（`05-场景功能/{终端}/pages/{页面名}/wireframe.md`）中描述的交互元素？

## 异常路径检查

- [ ] 每个测试用例是否至少有 1 个异常路径？
- [ ] 异常路径是否标注了对应 spec 的异常处理？
- [ ] 异常路径是否覆盖了关键分支？
- [ ] 异常路径是否符合规则矩阵（`05-场景功能/models/`）中的边界条件？
- [ ] 是否覆盖了 API 接口视图（`05-场景功能/{终端}/api-view.md`）中的错误码场景？

## 格式检查

- [ ] 文件命名是否符合 `test-{角色}-{场景}.md` 格式？
- [ ] 多终端项目是否按终端拆目录（`cases/{终端}/`）？
- [ ] 测试身份是否引用了 L3 的具体角色和环境描述？
- [ ] 期望结果是否可观察、可验证？
- [ ] index.md 是否包含测试矩阵、覆盖率矩阵和断链汇总三个部分？

## 写入前

以上所有检查项必须通过。任何一项不通过都必须补全后才能写入文件。
