# 测试用例方法论

## 核心方法：链式验证法

测试用例不是"功能清单"，而是**用户旅程**。每一步的"期望结果"必须是下一步的"前置条件"。

### 链式衔接示意

```
步骤1：打开首页 → 看到产品列表（含10个产品）     ✅ 链接到步骤2
步骤2：点击"新建报价" → 进入报价编辑页（可从列表选产品）  ✅ 链接到步骤3
步骤3：选择3个产品 → 页面显示已选产品和价格输入框    ✅ 链接到步骤4
步骤4：输入价格 → 底部出现"生成报价单"按钮          ✅ 链接到步骤5
步骤5：点击"生成报价单" → 显示报价预览和分享按钮    ✅ 完成目标
```

### 断链示例

```
步骤3：选择3个产品 → 页面显示已选产品              ⚠️ 没有价格输入框
步骤4：想输入价格 → ❌ 找不到输入框 → 断链！
```

断链 = 功能缺口，必须记录。

## 核心方法：覆盖率矩阵

### 构建方法

1. 遍历 L5 所有 spec 文件（`05-场景功能/{终端}/pages/{页面名}/spec.md`），收集到"spec 清单"
2. 遍历所有生成的测试用例，收集每个步骤引用的 spec
3. 交叉对比：spec 清单中有哪些没被引用 = 缺口

### 矩阵维度

- **按终端**：每个终端的 spec 是否都有测试覆盖
- **按模块**：每个模块的 API（从 `05-场景功能/{终端}/api-view.md` 提取）是否都被测试调用过

## 核心方法：异常路径挖掘

每个正常步骤问自己三个问题：

1. **做错了**：用户操作失误会怎样？（如选了 0 个产品点生成）
2. **做不了**：外部条件不满足会怎样？（如网络断了、数据被删了）
3. **做过了**：操作超过限制会怎样？（如选了 100 个产品）

### 异常路径来源

- spec 中的异常处理描述
- 状态机中的异常状态转移（`05-场景功能/models/` 中的状态机图）
- 规则矩阵中的边界条件（`05-场景功能/models/` 中的规则矩阵）

## 核心方法：多终端隔离

同一用户在不同终端的测试用例**完全独立**：
- 不能假设用户知道另一个终端有某个功能
- 不能假设用户在另一个终端上做过某个操作
- 不同终端的测试用例可以有不同的前置条件
- 不同终端的页面流程（`05-场景功能/{终端}/page-flow.md`）独立走查

## 常见误区

1. **写成功能清单** → 测试用例是"用户旅程"，不是"功能点列表"。必须按操作顺序串起来。
2. **忽略前置条件** → 每个测试用例开始前，系统必须处于确定的状态。
3. **期望结果不可验证** → "系统正常"不是期望结果。"页面显示 3 个已选产品，总价 ¥89.7"才是。
4. **跳过异常路径** → 只写正常流程的测试用例没有意义，异常才是最容易出问题的地方。
5. **覆盖率矩阵只报喜不报忧** → 矩阵的价值在于暴露缺口，不只是列出覆盖了的。
6. **忽略页面原型** → `wireframe.md` 包含页面布局信息，测试步骤中的"在哪"应与原型一致。
7. **忽略状态机约束** → 测试步骤中的状态流转必须符合 `models/` 中状态机的合法转移。
