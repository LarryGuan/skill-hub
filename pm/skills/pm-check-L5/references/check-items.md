# L5 检查清单

## 整体检查（28 项）

### A. 完整性（10 项）

#### A1. L4 场景覆盖
- **判定标准**：每个 L4 核心场景都有对应页面+功能规格 or 合理排除说明
- **检查方法**：从 L4 场景列表提取所有核心场景，逐个在 L5 页面和功能规格中查找对应；无对应的场景应有排除说明
- **通过**：每个 L4 核心场景在 L5 中都有页面+功能覆盖，或有合理的排除说明（如"该场景为人工流程，不涉及系统实现"）
- **不通过**：有核心场景在 L5 中无任何对应，也无排除说明

#### A2. L3 主体覆盖
- **判定标准**：所有 L3 主体都有对应终端的功能覆盖
- **检查方法**：从 L3 画像总览提取所有主体名，逐个在 L5 各终端的功能架构中查找覆盖
- **通过**：每个 L3 主体至少在一个终端有功能覆盖
- **不通过**：有 L3 主体在所有终端的功能架构中都没有出现

#### A3. 类图细化完整
- **判定标准**：覆盖所有场景中的实体，有字段、关系、场景对应
- **检查方法**：从 L4 场景中提取所有涉及的业务实体，逐个在 class.md 中查找；每个实体应有属性列表、关系标注、涉及场景标注
- **通过**：所有场景涉及的实体都在类图中有完整定义（字段+关系+场景对应）
- **不通过**：有场景中出现的实体在类图中缺失，或实体缺少字段/关系/场景标注

#### A4. 状态机完整
- **判定标准**：每个有状态变化的实体都有状态机图
- **检查方法**：从 class.md 中识别有"状态"字段的实体，检查 models/states/ 下是否有对应的状态机文件；每个状态机应包含：所有状态、合法转换、触发条件、异常转换
- **通过**：所有有状态变化的实体都有完整的状态机图，异常转换已覆盖
- **不通过**：有状态实体缺少状态机，或状态机缺少异常转换路径

#### A5. 规则矩阵完整
- **判定标准**：覆盖所有异常场景的边界条件，每条规则有来源场景
- **检查方法**：从 L4 场景中提取所有异常/边界条件，在 models/rules/ 中逐条查找对应规则；每条规则应有"来源场景"标注
- **通过**：所有异常场景的边界条件都有规则，每条规则标注了来源场景
- **不通过**：有异常场景缺少规则，或有规则未标注来源场景

#### A6. 终端目录完整
- **判定标准**：每个终端有 page-flow.md + api-view.md + pages/
- **检查方法**：从 L2 终端分工表提取所有终端，检查 05-场景功能/ 下每个终端目录的结构
- **通过**：每个终端目录都包含 page-flow.md + api-view.md + pages/ 子目录
- **不通过**：有终端缺少任一必要文件或目录

#### A7. 页面规格完整
- **判定标准**：每个页面有 spec.md + wireframe.md
- **检查方法**：遍历每个终端的 pages/ 子目录，检查每个页面目录是否包含 spec.md 和 wireframe.md
- **通过**：每个页面目录都包含 spec.md + wireframe.md
- **不通过**：有页面缺少 spec.md 或 wireframe.md

#### A8. 页面规格五要素
- **判定标准**：每个页面 spec 包含交互行为+规则落地+数据绑定+状态影响+场景回溯
- **检查方法**：打开每个页面的 spec.md，检查是否包含五个要素的对应章节或内容
- **通过**：每个 spec 都包含五个要素
- **不通过**：有 spec 缺少任一要素

#### A9. API 接口覆盖
- **判定标准**：覆盖所有页面的数据需求，每个接口含路径+方法+参数+响应+状态码
- **检查方法**：从各终端页面规格中提取数据需求（展示+提交），在对应终端的 api-view.md 中查找覆盖；每个接口应包含完整定义
- **通过**：所有页面的数据需求都有对应 API 接口，且接口定义完整
- **不通过**：有页面数据需求缺少对应接口，或接口定义不完整

#### A10. plan.md 进度完整
- **判定标准**：所有子步骤标记完成，4 个检查点都有记录
- **检查方法**：读取 plan.md，检查三阶段所有子步骤的状态标记，以及检查点 A/B/C/D 是否都有记录
- **通过**：所有子步骤标记为完成，4 个检查点都有记录内容
- **不通过**：有子步骤未标记完成，或检查点缺失

- **不通过**：CLAUDE.md 被 L5 修改过

- **不通过**：缺少必要目录或文件

### C. 功能质量（8 项）

#### C1. 四步法执行
- **判定标准**：功能不是 1:1 场景映射，有多个场景复用同一功能
- **检查方法**：在 index.md 的功能架构中，检查是否有功能被多个场景引用；统计功能数与场景数的比例
- **通过**：存在明显的功能复用（同一功能被 2+ 场景引用），功能数 < 场景数
- **不通过**：功能和场景 1:1 对应，没有复用，说明四步法未正确执行

#### C2. 高内聚低耦合
- **判定标准**：模块内功能相关，模块间依赖最小化
- **检查方法**：检查功能架构中的模块划分，验证模块内功能是否围绕同一业务概念组织，模块间是否有不必要的依赖
- **通过**：模块划分清晰，模块内功能高相关，模块间依赖最小
- **不通过**：存在跨模块的强耦合，或模块内功能不相关

#### C3. 功能命名规范
- **判定标准**：动词+名词，无虚词
- **检查方法**：逐个检查功能架构中的功能命名
- **通过**：所有功能命名遵循"动词+名词"格式，无虚词（如"智能优化""便捷操作"）
- **不通过**：有功能使用虚词命名（如"智能处理""自动优化"），无法从名称理解功能

#### C4. 功能可测试
- **判定标准**：每个功能点能写出测试用例
- **检查方法**：逐个检查功能规格中的功能点描述，判断是否有明确的输入/输出/条件可用于测试
- **通过**：每个功能点都有明确的可测试描述
- **不通过**：有功能点描述模糊，无法据此编写测试用例

#### C5. 模块编号+隐喻
- **判定标准**：前缀+序号统一，每个模块有隐喻式责任描述
- **检查方法**：检查功能架构中模块的编号规则和隐喻描述
- **通过**：所有模块编号统一（如 MOD-01、MOD-02），每个模块有隐喻式责任描述（如"订单指挥官"）
- **不通过**：编号不统一，或模块缺少隐喻描述

#### C6. spec 引用原则
- **判定标准**：spec 不重复定义领域知识，只引用+页面级交互补充
- **检查方法**：检查页面 spec.md 中的规则描述，是否有对 models/ 的引用（如引用规则矩阵的规则ID），而不是重新定义规则逻辑
- **通过**：spec 中规则部分引用 models/ 的定义，只补充页面级交互细节
- **不通过**：spec 中重复定义了 models/ 中已有的规则逻辑

#### C7. 功能架构图质量
- **判定标准**：分层排列+带标签数据流+外部系统交互
- **检查方法**：检查 index.md 中的功能架构图（Mermaid 格式）
- **通过**：架构图分层清晰（如展示层/业务层/数据层），数据流有标签，外部系统交互有标注
- **不通过**：架构图缺少分层，数据流无标签，或遗漏外部系统交互

#### C8. 场景覆盖对照
- **判定标准**：L4 场景→页面+API 映射表完整
- **检查方法**：检查 index.md 中是否有场景覆盖对照表，逐条验证每个场景都有页面+API 的映射
- **通过**：有完整的映射表，每个场景都有对应页面和 API
- **不通过**：缺少映射表，或映射表中有场景未映射

### D. 回溯验证（5 项）

#### D1. L2 终端分工一致
- **判定标准**：每个终端的页面差异与 L2 终端分工表一致
- **检查方法**：对比 L2 终端分工表中各终端的职责定位，与 L5 各终端的实际页面和功能是否匹配
- **通过**：各终端的功能和页面与 L2 终端分工表的职责定位一致
- **不通过**：有终端的页面/功能超出了 L2 分工范围，或 L2 分工要求的功能未实现

#### D2. L3 痛点覆盖
- **判定标准**：每个核心痛点有功能解决方案
- **检查方法**：从 L3 痛点列表提取核心痛点，逐个在 L5 功能架构中查找是否有功能直接或间接解决
- **通过**：每个核心痛点都有对应功能解决（在功能描述或 spec 中有体现）
- **不通过**：有核心痛点在 L5 功能中没有对应解决方案

#### D3. 跨终端数据一致
- **判定标准**：同一实体在不同终端 API 中的字段定义不矛盾
- **检查方法**：选取核心实体，对比各终端 api-view.md 中该实体相关接口的请求/响应字段
- **通过**：同一实体在不同终端 API 中的字段定义一致（名称、类型、约束相同）
- **不通过**：同一实体在不同终端 API 中字段定义有矛盾

#### D4. 运行闭环验证
- **判定标准**：每个核心实体有完整的创建→流转→消费路径
- **检查方法**：选取核心实体，追踪其在页面流程和 API 中的完整生命周期
- **通过**：每个核心实体都有完整的创建、状态流转、最终消费/归档路径
- **不通过**：有核心实体的生命周期不完整（如只有创建没有消费）

#### D5. 价值闭环验证
- **判定标准**：L1 验收标准有对应功能支撑，无过度设计功能
- **检查方法**：从 L1 验收标准逐条在 L5 功能中查找支撑；反向检查是否有功能无法追溯到任何验收标准
- **通过**：每条 L1 验收标准都有功能支撑，且无过度设计的功能
- **不通过**：有验收标准缺少功能支撑，或存在无法追溯到验收标准的功能

### 阶段二（终端交付，8 项）

#### P2-1. 功能架构四步法
- **判定标准**：有场景分析→能力抽象→模块设计→功能点细化过程
- **检查方法**：检查本终端的功能架构，是否体现了四步法的推导过程
- **通过**：功能架构中能看到从场景到能力的抽象、模块的组织、功能的细化
- **不通过**：功能架构缺少推导过程，直接列出了功能清单

#### P2-2. 功能复用性
- **判定标准**：至少有功能被多个场景复用
- **检查方法**：统计本终端功能和场景的映射关系，检查是否有功能被多个场景引用
- **通过**：存在被多个场景复用的功能
- **不通过**：功能和场景 1:1 对应，无复用

#### P2-3. 页面流程覆盖
- **判定标准**：所有 L4 场景的系统触点在页面流程中有节点
- **检查方法**：从 L4 本终端场景中提取系统触点，在 page-flow.md 中查找对应页面节点
- **通过**：每个系统触点在页面流程中都有对应节点
- **不通过**：有系统触点在页面流程中无对应节点

#### P2-4. spec 五要素完整
- **判定标准**：交互+规则+数据+状态+场景回溯
- **检查方法**：逐个检查本终端 pages/ 下每个页面的 spec.md，验证五个要素
- **通过**：每个 spec 都包含五个要素
- **不通过**：有 spec 缺少要素

#### P2-5. spec 不重复定义
- **判定标准**：引用 models/ 的 ID，不重复写规则逻辑
- **检查方法**：检查本终端各页面 spec 中的规则描述，是否有对 models/ 的引用
- **通过**：spec 中引用了 models/ 的规则 ID，只补充页面级交互细节
- **不通过**：spec 中重新定义了 models/ 中已有的规则逻辑

#### P2-6. 原型标注完整
- **判定标准**：交互元素+数据区域标注
- **检查方法**：检查本终端各页面 wireframe.md 中的标注
- **通过**：每个原型都有交互元素标注（如按钮、输入框）和数据区域标注（如列表、表单）
- **不通过**：有原型缺少交互元素或数据区域标注

#### P2-7. 痛点覆盖（本终端）
- **判定标准**：本终端相关痛点有功能解决
- **检查方法**：从 L3 痛点列表中筛选与本终端相关的痛点，在本终端功能中查找解决方案
- **通过**：本终端相关的核心痛点都有功能解决
- **不通过**：有本终端相关痛点无功能解决

#### P2-8. plan.md 阶段二进度
- **判定标准**：功能架构/页面流程/规格/原型/检查点C 标记完成
- **检查方法**：读取 plan.md，检查阶段二各子步骤和检查点 C 的状态
- **通过**：阶段二所有子步骤和检查点 C 都标记为完成
- **不通过**：有子步骤或检查点未标记完成

### 阶段三（API层，4 项）

#### P3-1. API 覆盖页面需求
- **判定标准**：每个页面的数据展示+提交需求都有对应接口
- **检查方法**：从本终端各页面规格中提取数据需求，在 api-view.md 中查找对应接口
- **通过**：每个页面的数据展示和提交需求都有对应 API 接口
- **不通过**：有页面的数据需求缺少对应接口

#### P3-2. API 引用实体
- **判定标准**：请求/响应字段引用 class.md 的实体字段
- **检查方法**：检查 api-view.md 中的接口定义，字段是否引用了 class.md 中的实体字段名
- **通过**：API 的请求/响应字段名与 class.md 中的实体字段定义一致
- **不通过**：有 API 字段名与 class.md 中的定义不一致，或使用了 class.md 中不存在的字段

#### P3-3. API 触发状态转换
